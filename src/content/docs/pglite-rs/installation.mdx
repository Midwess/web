# Installation

pglite-rs is published as the `pglite-rs` crate on [crates.io](https://crates.io/crates/pglite-rs). The engine ships as a prebuilt static library that the build script resolves for your target — from the bundled per-target artifact crate, a local cache, or a network download. No Postgres server, no Docker, no system packages.

## Install

```sh
cargo add pglite-rs
```

This pulls in `pglite-rs-sys` (the FFI layer, `links = "pglite"`). When you install from crates.io, the engine rides along inside the bundled per-target artifact crate as `libpglite.a.zst`, which `pglite-sys/build.rs` zstd-decompresses into `OUT_DIR` and links — no network needed. The runtime payload (`pglite-runtime.tar`, which carries `share/postgresql`) is resolved separately by `pglite/build.rs` and embedded into the library via `include_bytes!`. No C headers are involved — the FFI is hand-written in `pglite-sys/src/lib.rs` with no bindgen.

If no artifact crate is available for your target, the build script downloads a per-target tarball (`pglite-<target>.tar.gz`) of `libpglite.a` and sha256-verifies it against the release. Downloaded artifacts are cached at `~/.cache/pglite-rs/<ENGINE_TAG>/<target-triple>/<variant>` (the `<variant>` subdir is `icu` when the `icu` feature is on, empty otherwise). The current `ENGINE_TAG` is `engine-06c837c6a303-p3`.

> **Heads up:** the **package** is `pglite-rs`; the **library** is `pglite` (`use pglite::PGlite`).

```rust
use pglite::PGlite;
```

### How `build.rs` finds `libpglite.a`

The static library is resolved through an ordered precedence chain. The first match wins, so `PGLITE_LIB_DIR` always overrides the bundled artifact crate, and the network download is the last resort.

## Features

Pick the modes you need at build time.

| Cargo feature | What it adds |
|---|---|
| *(none — base)* | in-process engine, single session, initdb-on-open, `exec` / `query` / `transaction`, dump/restore |
| `socket` *(default)* | `PGlite::unix_uri()` — internal unix-socket gateway so unmodified ORMs (SQLx, SeaORM, Diesel, tokio-postgres) talk to the engine. Unix-only; inert on Windows. |
| `pgvector` | the pgvector extension — `CREATE EXTENSION vector`, embedding columns, similarity search |
| `pgcrypto` | the pgcrypto extension — digests, encryption (needs OpenSSL at artifact-build time only, not in your crate) |
| `icu` | ICU engine variant — real Unicode collation via `locale_provider: Icu` (~+40MB, statically bundled) |
| `multiple-process` | `PGlite::open_multi_process` — child postmaster on a private unix socket with pooled connections for true concurrent sessions. Unix-only. |
| `replica` | `Replica::start(db, config)` — logical-replication consumer; streams committed changes from a remote Postgres into a local read-only PGlite |
| `full` | every feature above in one go |

```sh
cargo add pglite-rs --features full
# or pick what you need
cargo add pglite-rs --features "pgvector,multiple-process"
```

> **Cross-feature note:** A data directory initialized with `locale_provider: Icu` can only be reopened by `icu`-feature builds, and vice-versa for libc datadirs. Mixing the two on the same path fails to open.

## Requirements

- **Rust.** Stable toolchain, **1.85+** (edition 2021). No nightly required.
- **A linker / system toolchain at *build* time** — not a C compiler. The consuming build compiles **no C**: the native shim (`pglite_native.o`) is already precompiled inside the prebuilt `libpglite.a`, and `pglite-sys/build.rs` has no `cc` build-dependency. You only need your platform's standard linker to fold the static library into your binary. On macOS, install Xcode Command Line Tools (`xcode-select --install`). On Debian/Ubuntu, `apt install build-essential`. On Windows, the toolchain that ships with `rustup` is enough. (A full C toolchain is only needed if you compile the engine yourself — see [build from source](#optional-build-the-engine-from-source).)
- **Network access on first build** *only when no bundled artifact crate or cache is available* for your target, so the build script can download and sha256-verify the prebuilt tarball. Downloaded artifacts are cached under `~/.cache/pglite-rs/<ENGINE_TAG>/<target-triple>/<variant>`. Once cached — or when installing from crates.io, which ships the engine in-crate — builds work offline.

That's it. No Postgres server. No `libpq`. No system SQLite. The native engine is statically linked straight into your binary.

### What actually links into your binary

No C is compiled at your build, and no headers ship. The prebuilt static library — carrying the precompiled `pglite_native.o` — is linked whole-archive through the hand-written FFI layer, while the runtime tar is baked into the library at compile time.

## Build-script note for extensions and plpgsql

If you turn on `pgvector`, `pgcrypto`, or use `plpgsql`, the engine `dlopen`s extension `.so` / `.dylib` files at runtime, and those modules need to resolve engine symbols back into your binary. **You must export dynamic symbols from the consuming binary**, or you will get mysterious "undefined symbol" errors at first query.

    <DArrow dir="right" kind="accent" />
    <DBox kind="warn" badge={3} title="vector / pgcrypto" sub={[".so / .dylib needs engine", "symbols at load"]} />
  </DRow>
  <DArrow dir="left" kind="accent" dashed label="resolves symbols back into the binary" />
  <DBox kind="muted" title="Base feature set" sub="no extensions, no plpgsql → nothing is dlopen'd → no export flag needed" />
</DChart>

Add a `build.rs` to your crate with this content:

```rust title="build.rs"
fn main() {
    #[cfg(target_os = "macos")]
    println!("cargo:rustc-link-arg=-Wl,-export_dynamic");
    #[cfg(not(target_os = "macos"))]
    println!("cargo:rustc-link-arg=-Wl,--export-dynamic");
}
```

Then declare the build script in your `Cargo.toml`:

```toml title="Cargo.toml"
[package]
build = "build.rs"
```

For the base feature set (no extensions, no plpgsql) this is not required.

## Verify

Create a new project to confirm the toolchain, the prebuilt artifact resolution, and the in-process engine all line up:

```sh
cargo new pglite-check --bin
cd pglite-check
cargo add pglite-rs
```

Then in `src/main.rs`:

```rust title="src/main.rs"
use pglite::PGlite;

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let db = PGlite::open_temp().await?;
    db.exec("CREATE TABLE hello (id serial PRIMARY KEY, msg text)")
        .await?;
    db.exec("INSERT INTO hello (msg) VALUES ('world')").await?;
    let rows = db.query("SELECT id, msg FROM hello", &[]).await?;
    println!("got {} row(s): {:?}", rows.len(), rows[0].get::<&str>(1)?);
    db.close().await?;
    Ok(())
}
```

```sh
cargo run
```

Expected output:

```
got 1 row(s): "world"
```

The first build takes a few seconds (resolves and links the prebuilt artifact, decompressing the bundled `libpglite.a.zst` or downloading the tarball on a cache miss). Subsequent builds reuse the cached library.

## Optional: build the engine from source

The prebuilt static library is what `cargo build` resolves by default (bundled artifact crate, cache, or download). If you want to compile the patched Postgres yourself (for a target triple that doesn't have a prebuilt, or to audit the source), build the artifact once and point the build script at it via `PGLITE_LIB_DIR` — which sits first in the resolution order, so it overrides everything else:

```sh
# in the pglite-rs repo
./native/build-libpglite.sh     # needs clang, bison, flex, perl, make, zlib
PGLITE_LIB_DIR=/path/to/dir cargo test --workspace
```

`PGLITE_LIB_DIR` is read by `pglite-sys/build.rs` for `libpglite.a`; the same directory must also carry `pglite-runtime.tar` (resolved by `pglite/build.rs`) for a full local build.
