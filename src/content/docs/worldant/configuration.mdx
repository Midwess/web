---
title: Configuration
description: The proposed open node configuration boundary for Worldant.
---

Configuration is intentionally not stable yet. The active proposal replaces Worldant's
built-in token and actor model with a snapshot-pinned TypeScript configuration module.

The proposed root configuration is executable middleware:

```ts
// worldant.ts
import { middleware } from "@midwess/worldant"
import type { NodeConfig } from "worldant"

export default {
  middleware: [
    middleware(),
    async function inspect(request, context, next) {
      return next(request, {
        ...context,
        requestId: crypto.randomUUID(),
      })
    },
  ],
} satisfies NodeConfig
```

`worldant build` captures the module and its dependencies in the immutable `.dist`
snapshot. `worldant run` loads middleware only from that snapshot.

Worldant assigns no authentication or authorization meaning to middleware, headers, or
request context. az-wire custom headers remain unchanged. For request-bound application
SQL, Worldant makes the original header object available as transaction-local
`request.headers`; application-authored PostgreSQL functions and RLS policies decide how,
or whether, to use it.

The proposal removes predefined actors, caller roles, token formats, claims, and
public/private judgments. Third-party libraries can export middleware factories without
requiring changes to Worldant or az-wire.
