# `GET /q/{hash}/{version}`

Fetch a content-and-version-addressed snapshot. The URL is the content — the same `{hash}/{version}` always returns the same body, served straight from the in-process cache.

## Path

```
GET /q/{hash}/{version}
```

- `{hash}` — hex of the `CacheableQuery` fingerprint: a `DefaultHasher` over the normalized (sqlparser-canonicalized) statement string, set in `classify.rs`. Identical normalized statements share a hash.
- `{version}` — the LSN watermark returned by `VersionIndex::version_of` for this query's tables and filters at issue time. It is an LSN, not a sequential counter, and advances as upstream writes touch the relevant table or row.

## Response

```
HTTP/1.1 200 OK
Content-Type: application/json
ETag: 9a4f:42
Cache-Control: public, max-age=259200
```

```json
[{"id":7,"email":"alice@example.com"}]
```

The body is the JSON-array text from the `jsonb_agg` wrapper in `rows::wrap_json` — `coalesce(jsonb_agg(to_jsonb(_t)), '[]'::jsonb)::text` — returned compactly, exactly as pglite produced it. An empty result set is `[]`.

The response is safe to cache at the CDN edge for the 72-hour `max-age` window. The URL is the cache key; the body is content-addressed.

## Headers

| Header | Value | Why |
|---|---|---|
| `Content-Type` | `application/json` | result rows are JSON |
| `ETag` | `{hash}:{version}` | strong validator equal to the cache key string; the URL is the actual cache key |
| `Cache-Control` | `public, max-age=259200` | cacheable by every CDN, browser, and proxy for 72 hours |
| `Vary` | *(none — the URL is the key)* | the URL is the cache key; no header-based variation |

After the 72-hour `max-age` elapses, a CDN may revalidate against the strong `ETag` (`{hash}:{version}`) the server emits. Because the URL is content-and-version-addressed, the same body is still correct — a revalidation simply confirms the existing entry.

## Stale versions

If the version in the URL is older than the current `VersionIndex` value, the request still returns the frozen snapshot. The current request gets a new URL on the next call to `POST /query`.

This is the right behavior:

- A client that fetches `/q/9a4f/42` and caches the URL gets that body for the 72-hour cache window (and may revalidate after via the `ETag`).
- A subsequent `POST /query` for the same SQL returns the new version's URL (`/q/9a4f/43`), pointing the client to the new snapshot.
- The old URL never has to be invalidated because it is content-addressed; the old body is still correct for the moment it was issued.

There is no forced `If-None-Match` round-trip during the `max-age` window — the URL itself is the cache key.

## What `404` means

`/q/{hash}/{version}` returns a `404` only if the cache key `{hash}:{version}` is not present — because the `{hash}` was never produced, or because the local moka cache evicted the body under memory pressure. The body is exactly `{"name":"NotFound","message":"unknown cursor"}` — a hand-written JSON response, not a `CacheError::envelope` payload, and the name is literally `NotFound` (no `Error` suffix). The URL is content-addressed, so a valid `{hash}/{version}` should normally find its body; `404` indicates cache eviction, not a missing snapshot.

If you see `404` in normal operation, your `--cache-size-bytes` is too small for the working set of snapshot URLs. Increase it.

## Examples

```bash
# Follow a 303 from POST /query
curl -L -X POST http://127.0.0.1:8080/query \
     -H 'content-type: application/json' \
     -d '{"sql":"select id, email from users where id = 7"}'
# [{"id":7,"email":"alice@example.com"}]

# Fetch a known snapshot URL directly
curl -i http://127.0.0.1:8080/q/9a4f/42
# HTTP/1.1 200 OK
# Content-Type: application/json
# ETag: 9a4f:42
# Cache-Control: public, max-age=259200

# CDN-friendly: same body within the 72h window
curl http://127.0.0.1:8080/q/9a4f/42
```
