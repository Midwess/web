# Listen & notify

pglite-rs exposes Postgres `LISTEN` / `NOTIFY` end-to-end — both as SQL you can issue directly, and as a Rust API that registers a callback and parses the `NotificationResponse` frames out of the engine's wire traffic. Live queries ride the same surface: subscribe to a SQL statement, and the host re-runs it on subscribed table changes and pushes fresh rows to your callback.

## Basic listen / notify

`listen(channel, callback)` registers a closure against a `LISTEN` channel. A `NOTIFY` issued from any source — the same process, or another session in multi-process mode — wakes the callback.

```rust
use pglite::PGlite;

let db = PGlite::open("./mydata").await?;

let _token = db
    .listen("orders", |payload| {
        println!("orders channel: {payload}");
    })
    .await?;

db.exec("NOTIFY orders, 'order-42 placed'").await?;
db.exec("NOTIFY orders, 'order-99 placed'").await?;
// prints:
//   orders channel: order-42 placed
//   orders channel: order-99 placed
```

The callback is `Fn(&str) + Send + Sync + 'static`. In in-process mode, native delivery is **not** a background stream: a `NOTIFY` surfaces only when its `NotificationResponse` rides back in the response buffer of some `exec` / `query` roundtrip you issue. The self-`NOTIFY` example above works because the notification frame is included in the reply to that same `exec`. Those frames are parsed by `process_response`, which runs on **your** calling task — not on the engine thread, which only pumps wire bytes and never inspects notifications. So a blocking callback stalls the calling task, not the engine; keep it short — push to a channel and return, don't do blocking work inside.

Continuous, async delivery exists only on the two paths that have a dedicated reader: the socket gateway doorman (the in-process ORM path) and the multi-process `NotifyConn` reader thread, both covered below.

## Unlisten

Two ways to drop a subscription:

```rust
// by token — only this listener is removed; the LISTEN SQL stays if others hold the channel
db.unlisten_token("orders", token).await?;

// by channel — every subscription on "orders" is removed; the LISTEN SQL is sent
db.unlisten("orders").await?;
```

When the last subscription on a channel is dropped, the host sends an `UNLISTEN "<channel>"` SQL to the engine. If the channel still has at least one subscription, the SQL is not sent — the existing `LISTEN` keeps the engine's notification stream open. `unlisten(channel)` removes every callback on the channel and always sends `UNLISTEN`.

## Channel name rules

Channel handling has two layers, and they disagree on case. The **Rust listener map is case-insensitive** — the host lowercases the channel both when storing a callback and when dispatching an incoming notification, so `listen("Orders", ...)` and `listen("orders", ...)` share the same map entry. But the **engine-side `LISTEN` is case-sensitive**: the host sends the original name double-quoted, so `listen("Orders", ...)` issues `LISTEN "Orders"` and `listen("orders", ...)` issues `LISTEN "orders"` — two distinct engine channels.

What actually gets delivered is decided by the engine's case-sensitive `LISTEN`, not by the map. So a plain `NOTIFY orders` (which Postgres folds to lowercase `orders`) will **not** reach a session that ran `listen("Orders", ...)`:

```sql
NOTIFY "Orders", 'hello';   -- delivered to listen("Orders", ...)
NOTIFY orders, 'hello';     -- folds to lowercase "orders"; NOT delivered to listen("Orders", ...)
```

If a channel name contains special characters or mixed case, quote it in `NOTIFY` to match. The host quotes automatically on the `listen` side.

## Native SQL

`LISTEN` and `UNLISTEN` work as raw SQL, too. The host is not involved unless you also want the callback — calling raw `LISTEN foo` SQL works but nothing in Rust will see the notifications.

```rust
db.exec("LISTEN orders").await?;
db.exec("NOTIFY orders, 'hello'").await?;
db.exec("UNLISTEN orders").await?;
```

This is useful for diagnostics and for tasks that want to handle notifications in plpgsql / triggers.

## Delivery in each mode

A `NOTIFY` reaches a Rust callback by three different routes, but all three converge on the same shared, lowercased listener map and fan out from there. The crucial difference is **which thread drives delivery** — and whether delivery is continuous or piggybacked on a roundtrip you initiate.

## Live queries

A live query is a host-layer subscription on top of `LISTEN` / `NOTIFY`. The flow:

1. You give the host a SQL statement.
2. The host wraps it in a `VIEW`, introspects every base table the query references (a recursive CTE), installs a refcounted `AFTER INSERT OR UPDATE OR DELETE` statement-level trigger per table, and `LISTEN`s on a `table_change__<schemaoid>__<tableoid>` channel for each.
3. A committed write to any watched table fires `pg_notify` on its channel. The host catches it, debounces, re-runs `SELECT * FROM <view>`, and pushes the fresh `Vec<Row>` to your callback.

```rust
let live = db
    .live_query("SELECT id, name FROM users WHERE id > $1", &[&0i32], |rows| {
        for row in rows {
            let id: i32 = row.get(0).unwrap_or(0);
            let name: &str = row.get(1).unwrap_or("");
            println!("{id} {name}");
        }
    })
    .await?;

// somewhere else, or in another task:
db.exec("INSERT INTO users (name) VALUES ('carol')").await?;
// callback fires with [id=N, name="carol"]

// when you are done — tear it down explicitly:
live.unsubscribe().await?;
```

Live queries are entirely a host concept — they ride the same `LISTEN` / `NOTIFY` surface that the engine already exposes, so they work the same in-process and in multi-process mode.

**`LiveQuery` is not cleaned up on drop.** There is no `Drop` impl. Dropping the handle drops the wake channel sender, which stops the `pglite-live-refresh` thread — but it leaves the `VIEW`, the per-table triggers, and the `LISTEN` registrations (and their listener-map entries) in place. To tear all of that down you must call `LiveQuery::unsubscribe(self).await`, which `UNLISTEN`s each channel via `unlisten_token`, decrements (and drops on zero) the per-table triggers, and `DROP`s the view. Orphaned views left behind by a dropped-but-not-unsubscribed handle are only reclaimed on the next `open()`, via `sweep_live_views`.

## Multi-process: the NotifyConn

In multi-process mode, the pool is busy doing real work. `LISTEN` subscriptions are held by a dedicated **NotifyConn** that lives outside the pool:

- Created lazily on the first `listen()` or `live_query` call.
- Holds all `LISTEN` registrations in one place.
- A blocking reader thread (`pglite-notify`) streams `NotificationResponse` messages out of it and fans them to the Rust listener map.
- Live queries and `listen()` callbacks both read from the same map, so subscribers do not need a pool slot.

The NotifyConn is owned by the `PGlite` and is closed on `close()` / last drop. You do not interact with it directly.

## Error handling

A `listen()` call that targets a closed engine returns `Error::Closed`. A `NOTIFY` from SQL that fires for a non-subscribed channel is silently dropped on the engine side — no error, no callback.

In in-process mode, a native notification is only delivered when its `NotificationResponse` is piggybacked on the reply to an `exec` / `query` roundtrip, and `process_response` invokes your callback on the calling task. So if you `await` or block inside the callback, you stall **that task**, not the engine thread (which never sees notifications). The recommended pattern is to push to a `futures::channel::mpsc` channel from the callback and process the events on a separate task. On the socket-gateway and multi-process paths the same advice holds, except the callback runs on the doorman or `pglite-notify` reader thread respectively, so a slow callback stalls that reader.
