Worldant keeps the durable state machine inside your Node process and keeps socket/process ownership in the supervisor. The two communicate only when the app is launched under `worldant run`.

## Durable world

The library loads a napi addon and exposes the `@workflow/world` interface. Calls such as `runs.get`, `events.create`, `steps.list`, hooks, streams, and queue dispatch go through native bindings. Data is encoded through Worldant's codec before crossing the JS/native boundary.

When `install()` is used, Worldant also calls `setWorld(world)` and `world.start()`.

## Process supervision

The CLI starts your command with a control channel. When the JS package sees that channel, it:

- installs an HTTP request interceptor for `/__worldant/v1`;
- intercepts `http.Server.prototype.listen`;
- asks the app to listen on a private Unix socket;
- reports the intended host/port and private socket to the supervisor.

The supervisor then binds the public port and forwards bytes to the private socket.

## Recovery model

Workflow progress is stored in the world. On restart, the new process opens the same data directory and starts the world. Completed steps remain completed. Active runs can continue from persisted events.

## Network endpoints

If `createWorld({ net: true })` or `createWorld({ net: { ... } })` is used, Worldant serves `/__worldant/v1` either on the app's own HTTP server or an explicit port. That endpoint handles registry metadata, npm packuments, run creation, run status, streams, and hook resume.
