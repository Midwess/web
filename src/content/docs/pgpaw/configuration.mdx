# Configuration

Every CLI flag has a matching `UPPERCASE_SNAKE_CASE` environment variable. The flag wins when both are set. This page scans them all.

Every flag is a global `clap` option, so the same values apply to both `pgpaw init` and `pgpaw serve`. From the CLI they flow into `UpstreamConfig` / `ServerConfig`, then into `Di::init`, which fans them out to the subsystems they configure.

## HTTP

| Flag | Env | Default | Description |
|---|---|---|---|
| `--host` | `CACHE_HOST` | `127.0.0.1` | bind host |
| `--port` | `CACHE_PORT` | `8080` | bind port |
| `--data-dir` | `CACHE_DATA_DIR` | `./cache-data` | pglite-rs replica data directory |
| `--max-connections` | `CACHE_MAX_CONNECTIONS` | `8` | replica pool size (multi-process) |
| `--cache-size-bytes` | `CACHE_SIZE_BYTES` | `268_435_456` (256 MiB) | result-cache budget |
| `--cors-origin` | `CORS_ORIGIN` | *(none)* | allow browser cross-origin calls: an origin, a comma list, or `*`. Required when a browser app on another origin calls PgPaw directly |

`--max-connections` is the size of the pglite-rs multi-process pool that backs reads. Bigger pools cost more memory; the cache hit path does not need more than one. The pool size matters only for concurrent reads, and the in-process cache hits short-circuit the pool entirely.

`--cache-size-bytes` is the `moka` cache budget, weighed by each cached body's byte length. Default 256 MiB is comfortable for most workloads. Increase it if your eviction rate is high, decrease it if memory is tight.

`--cors-origin` is the single most common source of "the browser call works from curl but fails in the page". It maps directly to the actix CORS layer: leave it unset for the same-origin default, set `*` to allow any origin / method / header, or pass a comma-separated allowlist of origins (e.g. `https://app.example.com`) and the preflight goes through.

## Upstream Postgres

| Flag | Env | Default | Description |
|---|---|---|---|
| `--pg-host` | `UPSTREAM_HOST` | `127.0.0.1` | upstream hostname or IP |
| `--pg-port` | `UPSTREAM_PORT` | `5432` | upstream port |
| `--pg-user` | `UPSTREAM_USER` | `postgres` | upstream role with `REPLICATION` and `LOGIN` |
| `--pg-password` | `UPSTREAM_PASSWORD` | *(empty)* | upstream password |
| `--pg-database` | `UPSTREAM_DATABASE` | `postgres` | upstream database |
| `--publication` | `UPSTREAM_PUBLICATION` | `cache_server_pub` | logical-replication publication name |
| `--slot` | `UPSTREAM_SLOT` | `cache_server_slot` | replication slot name |
| `--sslmode` | `UPSTREAM_SSLMODE` | `disable` | one of `disable` / `prefer` / `require` / `verify-full` |

The `pgpaw init` command sets sensible defaults for `--publication` and `--slot`. If you customize the publication name on the upstream, pass the same name to `serve`.

`--sslmode` is parsed by `parse_sslmode` and forwarded into pglite-rs's `ReplicaConfig` for the upstream logical-replication connection that `serve` opens. Recognized values are `prefer`, `require`, and `verify-full` (also spelled `verify_full` / `verifyfull`); anything else — including the default `disable` — falls back to `Disable`. Note that `--sslmode` applies only to `serve`: the one-shot `pgpaw init` connects without TLS (`NoTls`), so the flag does not affect it.

## Authorization (JWT)

Set at most one verification key. With none, PgPaw is anonymous-only (public data only).

| Flag | Env | Default | Description |
|---|---|---|---|
| `--jwt-secret` | `JWT_SECRET` | *(none)* | HS256 shared secret |
| `--jwt-public-key` | `JWT_PUBLIC_KEY` | *(none)* | RS256/ES256 verification public key (PEM) |
| `--jwt-jwks-url` | `JWT_JWKS_URL` | *(none)* | JWKS endpoint — *planned*; use `--jwt-public-key` today |
| `--jwt-role-claim` | `JWT_ROLE_CLAIM` | `role` | JWT claim naming the Postgres role to run reads as |

`--jwt-jwks-url` is accepted as a flag but is not yet implemented — if it is the chosen key, `Verifier::build` returns a configuration error at startup. Use `--jwt-public-key` for asymmetric verification today.

See [Authorization](/pgpaw/authorization) for the full claims contract and policy writing guide.

## `pgpaw init`

`init` is a one-shot command — it does not run continuously and reads the same Postgres flags as `serve`. It also accepts the publication and slot names so it creates the publication matching the names `serve` will use. It connects with plain `tokio-postgres` (no TLS).

```bash
pgpaw init \
    --pg-host 127.0.0.1 --pg-user postgres --pg-password "$POSTGRES_PASSWORD" \
    --pg-database myapp \
    --publication cache_server_pub --slot cache_server_slot
```

`init` does three things:

1. Sets `wal_level=logical` if it is not already, and raises `max_wal_senders` and `max_replication_slots` to `10` if either is below `10`. These settings are restart-only — if any changes, PgPaw prints a notice telling you to restart Postgres to apply them.
2. Creates the publication (`--publication`, default `cache_server_pub`) covering all tables, if it does not already exist.
3. Installs the DDL event trigger `pglite_ddl_watch` on `ddl_command_end`, which calls the function `pglite_emit_ddl()`. That function emits a logical message (using pglite's `DDL_SIGNAL_PREFIX`) so the replica sees schema changes. If the role lacks the privilege to install it, `init` prints a warning and continues — online schema changes fall back to reactive recovery.

`init` does not precheck role privileges; it just runs each step. If the role cannot apply a setting or create the publication, that step warns and falls back rather than failing a privilege gate.

`init` is durable — the publication and trigger persist in the catalog. The replication slot is **not** created by `init`; it is created automatically when `serve` first starts the replica. Run `init` once per upstream.

## `pgpaw serve` vs `pgpaw init`

`init` is for upstream preparation (one-shot, no TLS). `serve` is the long-running process that serves HTTP. The two commands share the upstream flags but differ in what they do with the database.

| | `init` | `serve` |
|---|---|---|
| Sets `wal_level=logical` (+ senders/slots) | yes | no |
| Creates the publication | yes | no |
| Installs the DDL trigger | yes | no |
| Creates the replication slot | no | yes (at startup) |
| Opens the pglite-rs replica | no | yes |
| Binds the HTTP port | no | yes |
| Streams CDC to VersionIndex | no | yes |
| Idempotent | yes (safe to re-run; prints "already exists"/"already set" and skips) | yes (resumes from watermark) |

## On-disk paths

| Path | Lifetime | Contents |
|---|---|---|
| `<dataDir>/` (default `./cache-data/`) | Persistent across runs | pglite-rs replica cluster: tables, RLS, grants, `_pglite_replica` meta table |
| `<dataDir>/postmaster.pid` | Multi-process mode only | Standard Postgres lock file. Stale on SIGKILL — next boot handles stale-pid per Postgres' own rules |

`dataDir` is yours. There is no separate state directory, no Redis, no schema cache file. Remove the dir to start over.
