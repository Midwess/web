# `POST /query`

Run a read-only `SELECT`. Returns a `303 See Other` for public queries (pointing to the snapshot URL) or the rows inline for access-controlled queries.

## Request

```json
{ "sql": "select id, email from users where id = 7" }
```

Headers:

- `Authorization: Bearer <jwt>` — required for access-controlled queries, omitted for public ones
- `content-type: application/json`

The body is a single object with a `sql` field. Anything else returns a `400` — name `RejectedError` for classifier rejections (multi-statement, non-`SELECT`, writes, DDL, locking reads, side-effecting/volatile functions, non-replicated tables) or `ParseError` for SQL that does not parse. See [SQL classifier](/pgpaw/sql-classifier) for the full rules.

## Routing: one POST, three responses

A single `POST /query` resolves to one of three response shapes. The request first passes the halt check, then auth, then the classifier, then the public-vs-access-controlled verdict before branching.

## Public response — `303 See Other`

```
HTTP/1.1 303 See Other
Location: /q/9a4f/42
Cache-Control: no-store
```

The body is empty. The `Location` URL points at the versioned snapshot — fetch it (with `curl -L`, or follow the redirect in fetch / browser) to get the actual result.

The `303` keeps `POST` semantics: a `POST` body is non-cacheable, but the `Location` header on a `303` is cacheable. Browsers, fetch, and `curl -L` all follow the redirect to the snapshot URL.

## Access-controlled response — `200 OK`

```
HTTP/1.1 200 OK
Content-Type: application/json
Cache-Control: private, no-store
```

```json
[
  { "id": 7, "total": 4200 }
]
```

Access-controlled queries do not redirect. Rows are returned inline. The response is never cached and never CDN-served.

The classification happens once per query against the current replicated catalog. A query that is public today can become access-controlled if a downstream RLS change moves the table into the access-controlled bucket — re-classification is automatic when `Replica::security_version()` bumps (the verdict cache is keyed on it). See [Authorization](/pgpaw/authorization) for the public vs access-controlled split.

## The hash and the version

The `Location` URL has the shape `/q/{hash}/{version}` — a bare hash segment and a bare numeric LSN, no `v=` prefix.

- **Hash** is `DefaultHasher` over the normalized (re-serialized parsed) statement. Two byte-different SQL strings that parse to the same AST share a fingerprint; the hash is `format!("{:x}", query.fingerprint)`.
- **Version** is the `VersionIndex` value at lookup time. For a single-table query with an anchorable equality filter, it is `max(value-watermark, structural floor)`. For joins, multi-table queries, or queries without an anchorable filter, it falls back to the max per-table / structural LSN across all touched tables.

Two clients that fetch `/q/9a4f/42` always get the same body. A relevant upstream write bumps the version, so a new request gets a new URL. The old URL keeps serving its frozen snapshot for as long as the [`/q` resource](/pgpaw/api/snapshot) stays cached — `public, max-age=259200` (72 hours), by default.

For a `WHERE pk = literal` query, the anchor is `(table, pk, value)`. Writes to other rows do not bump the LSN for the queried value, so the URL stays valid. Joins and queries without anchorable equality filters fall back to per-table versioning.

## Examples

Public, anchored — same URL across writes to other rows:

```bash
curl -i -X POST http://127.0.0.1:8080/query \
     -H 'content-type: application/json' \
     -d '{"sql":"select id, email from users where id = 7"}'
# HTTP/1.1 303 See Other
# Location: /q/9a4f/42
# Cache-Control: no-store
```

Public, unanchored — version bumps on every write to the table:

```bash
curl -i -X POST http://127.0.0.1:8080/query \
     -H 'content-type: application/json' \
     -d '{"sql":"select id, status from orders where status = '\''paid'\''"}'
```

Access-controlled — rows inline, never cached:

```bash
curl -X POST http://127.0.0.1:8080/query \
     -H 'authorization: Bearer <jwt>' \
     -H 'content-type: application/json' \
     -d '{"sql":"select id, total from orders"}'
# [{"id":7,"total":4200}]
```
