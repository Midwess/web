# Installation

PgPaw is a single static binary. Install it with whichever toolchain you have, then run `pgpaw init` once against the upstream Postgres to set up the logical replication source.

## Install

```bash
# via Cargo (builds from source — needs a Rust toolchain)
cargo install pgpaw

# via npm (downloads a prebuilt binary for your platform)
npm install -g pgpaw
```

Prebuilt binaries are published for:

| Platform | Architecture | Rust target |
|---|---|---|
| Linux | `x86_64` (glibc) | `x86_64-unknown-linux-gnu` |
| macOS | Apple Silicon (`arm64`) | `aarch64-apple-darwin` |

Other platforms build from source via `cargo install pgpaw`. Cargo and npm are the only tools you need.

The npm install runs a `postinstall` script that resolves your `platform:arch` against the table above and downloads the matching prebuilt tarball directly from the project's GitHub releases (`v{version}/pgpaw-{target}.tar.gz`). If no prebuilt target matches your platform, it prints a message and tells you to `cargo install pgpaw` from source instead.

## Verify

```bash
pgpaw --version
pgpaw --help
```

You should see the subcommand list (`init`, `serve`) and the full set of flags.

A couple of cosmetic quirks worth knowing: the binary's built-in `--help` text and the `init`/`serve` console messages still self-identify as `cache-server` internally (e.g. "start the server with `cache-server serve`") — the published binary is named `pgpaw`, so keep using `pgpaw` to invoke it; the old name is harmless. The crate version (`0.2.5`) and the npm package version (`0.2.4`) can also diverge, so `--version` may not match the npm package version exactly.

## Requirements

- **A reachable upstream Postgres** with superuser access. PgPaw needs Postgres that supports logical replication (`wal_level=logical`, a server-level setting that requires a restart) and event triggers, and the privilege to run `ALTER SYSTEM`, `CREATE PUBLICATION`, and `CREATE EVENT TRIGGER`. All of this is done by `pgpaw init`.
- **For `cargo install`:** Rust 1.85+ on the build machine. The resulting binary is static and runs anywhere.
- **For `npm install`:** nothing extra. The `postinstall` script downloads the matching prebuilt binary from the project's GitHub releases for your OS/arch.
- **Disk:** the replica data dir lives at `./cache-data` by default (`--data-dir`), holding the embedded pglite replica plus your replicated data.
- **Network:** the replica connects to the upstream Postgres for logical replication. The PgPaw process binds an HTTP port for `serve`. No other inbound or outbound connections.

## One-time upstream setup: `pgpaw init`

Before serving queries, run `pgpaw init` against the upstream Postgres. It makes the following changes (all idempotent):

1. Sets `wal_level=logical` if it is not already. This is a restart-only setting — if `init` changes it, PgPaw prints a notice to restart Postgres before it takes effect.
2. Raises `max_wal_senders` and `max_replication_slots` to at least `10` (via `ALTER SYSTEM`) if they are lower. These are also restart-only, contributing to the same restart notice.
3. Creates the publication (`cache_server_pub` by default) `FOR ALL TABLES`, only if it does not already exist.
4. Installs a function `pglite_emit_ddl()` and an event trigger named `pglite_ddl_watch` firing on `ddl_command_end`, so the replica sees schema changes online.

The replication slot is **not** created by `init` — it is created automatically when the server starts.

```bash
pgpaw init \
    --pg-host 127.0.0.1 \
    --pg-user postgres \
    --pg-password "$POSTGRES_PASSWORD" \
    --pg-database myapp
```

After `init`, if it printed a notice about restart-only settings, restart Postgres. The publication and trigger are durable in the catalog — you only need to run `init` once per upstream.

## Then serve

```bash
pgpaw serve \
    --pg-host 127.0.0.1 \
    --pg-user postgres \
    --pg-password "$POSTGRES_PASSWORD" \
    --pg-database myapp \
    --port 8080
```

`serve` boots in order: it runs **preflight** validation against the upstream (requires `wal_level=logical`, `max_wal_senders` and `max_replication_slots` ≥ 1, and that the publication exists — otherwise it errors with remediation text and refuses to start). Note the asymmetry: `init` *applies* the value `10`, but the serve-time gate only requires ≥ 1. It then opens the multi-process pglite replica (`open_multi_process`), starts the replica (`Replica::start`), scans the schema (replicated tables, single-column PKs, `REPLICA IDENTITY FULL` tables), wires the CDC bridge (a dedicated OS thread named `cache-cdc`) into the version index and the live hub, builds the moka cache, the SQL classifier, and the JWT verifier, and finally binds the HTTP port.

## Health check

```bash
curl -s http://127.0.0.1:8080/healthz
```

`{"status":"ok","watermark":12345678}` once the replica is caught up. `{"status":"halted","reason":"…"}` if the applier halted (incompatible DDL, security reconcile failure, etc.).
