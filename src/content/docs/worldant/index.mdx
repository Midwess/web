# Worldant

Worldant is a [Vercel Workflow](https://github.com/vercel/workflow) world that runs **inside your Node.js process** and persists to embedded SQLite. No external database, no separate service. A supervisor CLI owns the public TCP port, scales the host to zero when idle, and wakes it on the next request.

It is built for **agentic and long-running workloads** that need durable execution without standing up Postgres, Redis, or a managed workflow service.

## How the pieces fit

Worldant ships as two packages around one durable core. The library is the durable World, loaded into your app's own process; the supervisor is a standalone binary that owns the public port and forwards every byte to the app over a private unix socket. The supervisor never opens the World or the database — durability lives entirely in the library.

## Three ways to call a workflow

A durable workflow is only useful once something can reach it. Worldant's goal is to expose one workflow through every calling surface you need — the workflow underneath never changes, only how it is reached.

| Surface | What it means | Status |
|---|---|---|
| **API** | Your app's HTTP routes (NestJS / Express / Next) trigger workflows. The supervisor owns the public port and forwards every byte to your app. | **Shipped** |
| **Remote function** | A deployed world serves itself as a live npm registry, so any consumer installs a generated, typed RPC client and calls your workflows like local functions ([same-origin](/worldant/remote-functions/same-origin) / [cross-origin](/worldant/remote-functions/cross-origin)). | **Shipped** |
| **MCP** | The same workflows surfaced as [Model Context Protocol](https://modelcontextprotocol.io) tools, so an agent can invoke them. | Roadmap |

## Features

- **Embedded storage.** Runs, steps, events, hooks, queues, and streams in a single SQLite file. WAL mode, single writer, reader pool. No external database.
- **Drop-in for `@workflow/world`.** Same `runs` / `steps` / `events` / `hooks` / `streams` / `queue` API. Workflows defined with `"use workflow"` and `"use step"` work unchanged.
- **In-process dispatch.** No worker to run, no separate Node service. The world lives where your app lives.
- **Scale-to-zero supervisor.** A `worldant` CLI wraps your app, owns the public TCP port, and stops the process when idle. The next request restarts it and resumes the log.
- **Hooks and streams.** Built-in support for `createHook` / `await hook` (durable external waits) and `getWritable` (durable byte streams).
- **Retries with `retryAfter`.** `RetryableError` / `FatalError` flow into the world's retry scheduler. The next attempt runs the same step body, with results memoized.
- **Cross-process exactly-once.** A durable step that already committed is never re-executed, even after a crash. In-flight runs are reenqueued and resumed when `world.start()` runs.
- **Child workflows.** `start(childWorkflow, args)` from `workflow/api` spawns a parallel run that the world tracks independently.
- **In-memory test mode.** Omit `dataDir` to get an ephemeral, in-memory database for tests.

## How it fits

Vercel Workflow is a two-part system:

- **`@workflow/core`** owns the workflow model: the `"use workflow"` / `"use step"` compiler transform, the runtime that replays workflow bodies, the determinism guard, the hook / stream / queue primitives that user code calls. It is the part your code interacts with.
- **`@workflow/world`** owns the storage: every run, step, event, hook, and chunk of stream data is persisted through it. It is the part that talks to a database.

Worldant replaces the **`@workflow/world`** layer. You write code against `@workflow/core` exactly the same way; the only thing that changes is which world you install at boot.

```ts
// Before (Vercel default)
import { start } from 'workflow/api';
// @workflow/core is wired up to the Vercel-hosted world automatically

// With worldant — at your app's entry, before it serves
import { install } from '@midwess/worldant';
import { start } from 'workflow/api';

await install(); // durable World on embedded SQLite, registered into @workflow/core
```

`install()` is the entire integration: it calls `createWorld()`, registers the world with `@workflow/core` via `setWorld()`, then runs `world.start()` (which reenqueues any in-flight runs and starts the in-process queue worker). Importing the package alone does **not** wire the world in — the only thing module import does on its own is arm the supervisor's `listen()` hijack (`connectSupervisor`). The rest of your workflow code is unchanged. The world underneath is now a Rust-backed SQLite file instead of a Vercel-hosted service.

## Boot order, from import to serving

Under the supervisor, the app comes up in a fixed sequence. Importing the library arms the `listen()` hijack; `install()` opens the durable World, announces the app to the supervisor, registers it with `@workflow/core`, and starts the queue worker; then `app.listen()` is transparently relocated onto a private unix socket so the supervisor can own the public port.
