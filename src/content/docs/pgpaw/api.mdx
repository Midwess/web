# API

Four endpoints. All bodies are JSON, all errors are JSON. The `GET /q/{hash}/{version}` snapshot resource is CDN-cacheable for 72 hours; the `303` redirect that points at it is `Cache-Control: no-store`.

| Endpoint | Mode | Purpose |
|---|---|---|
| [`POST /query`](/pgpaw/api/query) | snapshot | Run a read-only `SELECT`, return a `303` redirect to the snapshot URL |
| [`POST /query?live=true`](/pgpaw/api/live) | SSE | Run, then open an SSE stream of row-level deltas |
| [`GET /q/{hash}/{version}`](/pgpaw/api/snapshot) | fetch | Fetch a CDN-cacheable snapshot (72h window) |
| `GET /healthz` | health | Liveness + replica watermark (`ok` / `halted`) |

These are the only four routes — there is no `/metrics`, `/version`, or anything else.

`POST /query` is the entry point. It returns either a `303 See Other` to a snapshot URL (public queries) or the rows inline (access-controlled). The snapshot URL is content + version-addressed: a data change yields a new version, so a new URL.

`POST /query?live=true` is the same entry point with `live=true`. It opens an SSE stream after the initial snapshot, emitting row-level deltas on every relevant upstream commit.

`GET /q/{hash}/{version}` is the snapshot fetch. Because the version is part of the URL, the body for a given `{hash}/{version}` is stable, and the resource is cacheable for 72 hours (`Cache-Control: public, max-age=259200`). The response also carries an `ETag` (`{hash}:{version}`), so conditional `If-None-Match` revalidation is supported.

`GET /healthz` is the liveness check. Returns `{ status, watermark }` while the replica is keeping up; `{ status: "halted", reason }` if the applier ran into an unrecoverable error.

## Error envelope

Every error response is JSON:

```json
{ "name": "RejectedError", "message": "table `secrets` is not available in this cache (not replicated)" }
```

| Status | `name` | When |
|---|---|---|
| `400` | `RejectedError` | The [classifier](/pgpaw/sql-classifier) rejects the query (writes / DDL / volatile funcs / multi-statement / non-replicated table / locking reads / `SELECT INTO`) |
| `400` | `ParseError` | SQL does not parse |
| `401` | `UnauthorizedError` | Query is access-controlled and the bearer token is missing, invalid, or expired — **or** a bearer token is supplied but no JWT verifier is configured |
| `403` | `ForbiddenError` | An upstream/RLS database denial during execution — mapped from sqlstates `42501` (insufficient privilege), `42704` (undefined object), `28000` (invalid authorization) |
| `404` | `NotFound` | `/q/{hash}/{version}` does not match a known snapshot (literal hand-written body, not a `CacheError` variant) |
| `503` | `HaltedError` | Replica is halted; reads cannot be served |
| `500` | *unmapped* | Any error not mapped above — `PgliteError` / `UpstreamError` / `ConfigError` / `CacheError` / `IoError`, each carrying its own `name()` |

The `name` is stable across releases and safe to switch on client-side. The `message` is for humans. With no `Authorization` header on a public query, the request is allowed and runs anonymously.

## CORS

Browsers on a different origin than PgPaw need a `CORS_ORIGIN` (or `--cors-origin`) — pass an origin, a comma list, or `*` to allow cross-origin `POST /query` and SSE:

```bash
pgpaw serve --cors-origin "https://app.example.com,https://staging.example.com"
```

When `CORS_ORIGIN` is unset, PgPaw uses `Cors::default()`. Same-origin requests work as-is; only cross-origin browser calls are blocked until you set an origin, a comma list, or `*`.
