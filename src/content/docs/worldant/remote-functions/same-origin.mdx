# Same-origin remote functions (Next.js)

When your frontend lives in the same Next.js project as your workflows, `withWorldant` handles everything: it scans your source files at build time, generates a typed client, and a rewrite rule makes the browser call its own origin — no CORS required.

## Installation

```sh
npm install @midwess/worldant worldant-client
```

## 1. Wrap `next.config.ts`

```ts
// next.config.ts
import { withWorkflow } from 'workflow/next';
import { withWorldant } from '@midwess/worldant/next';

const WORLD_PORT = Number(process.env.WORLDANT_NET_PORT ?? 4001);

const nextConfig = {
  async rewrites() {
    return [
      {
        source: '/__worldant/v1/:path*',
        destination: `http://127.0.0.1:${WORLD_PORT}/__worldant/v1/:path*`,
      },
    ];
  },
};

export default withWorkflow(withWorldant(nextConfig, { client: true }));
```

`withWorldant(..., { client: true })` tells the build to scan `"use workflow"` files and write `worldant/client.ts` to your project root on every `next dev` / `next build`.

## 2. Start the world in `instrumentation.ts`

```ts
// instrumentation.ts
export async function register() {
  if (process.env.NEXT_RUNTIME === 'edge') return;
  const { createWorld } = await import('@midwess/worldant');
  const port = Number(process.env.WORLDANT_NET_PORT ?? 4001);
  await createWorld({ id: 'orders', net: { port } });
}
```

This starts the world in the Node.js runtime when the Next.js server boots. The `net: { port }` option starts a dedicated HTTP server for the world on the given port.

## 3. Add a path alias

The generated client lives at `./worldant/client.ts` in your project root. Add a path alias so imports look like a package, not a relative path:

```json
// tsconfig.json
{
  "compilerOptions": {
    "paths": {
      "worldant/client": ["./worldant/client"]
    }
  }
}
```

> Run `next dev` once first — `worldant/client.ts` must exist before TypeScript resolves the alias. After that, your editor and type checker see the generated types normally.

## 4. Call from a component

```tsx
// app/checkout-button.tsx
'use client';
import { placeOrder, trackOrder } from 'worldant/client';

export function CheckoutButton({ order }) {
  async function handleClick() {
    const receipt = await placeOrder(order);
    console.log('Order placed:', receipt.orderId, receipt.totalCents);

    for await (const status of trackOrder(order.id)) {
      console.log('Status:', status.stage);
    }
  }
  return <button onClick={handleClick}>Place order</button>;
}
```

The browser fetches `/__worldant/v1/runs` — its own origin, proxied server-side by Next.js to the world port. No CORS headers needed.

## Run IDs and idempotency (`CallOptions`)

Every generated wrapper accepts an optional second argument `opts?: CallOptions`:

```ts
import type { CallOptions } from 'worldant-client';

// opts is always optional — plain calls work unchanged
const receipt = await placeOrder(order);
const receipt = await placeOrder(order, { runId: '...' });
const receipt = await placeOrder(order, { scopeKey: 'order:123' });
```

### Auto-idempotent by default (no opts)

Every call is idempotent by default. The client auto-generates a conformant `wrun_<ULID>` per call, in memory, scoped to that call. If the network drops or worldant restarts mid-flight, retrying sends the same `runId` — the world resolves the existing run and returns its result. **No duplicate runs.**

```tsx
// Zero setup — connection drops, worldant restarts: still exactly one run
const receipt = await placeOrder(order);
```

### Explicit `runId`

Supply your own id to control dedup from outside:

```tsx
import { generateRunId } from 'worldant-client';

// Mint once, use across retries from any surface (browser, server action, …)
const runId = generateRunId();
const receipt = await placeOrder(order, { runId });
```

`generateRunId()` is also exported from the generated `worldant/client.ts` so you don't need a separate import:

```ts
import { placeOrder, generateRunId } from 'worldant/client';
```

### `scopeKey` — survive a page reload

The auto-id is in-memory. When the user might leave the page and return while the run is still in progress, use `scopeKey`:

```tsx
'use client';
import { placeOrder } from 'worldant/client';

export function CheckoutButton({ cartId, order }) {
  async function handleClick() {
    // The id is stored in localStorage under 'worldant:run:order:<cartId>'.
    // A reload resumes the existing run instead of starting a new one.
    const receipt = await placeOrder(order, { scopeKey: `order:${cartId}` });
    console.log('Placed:', receipt.orderId);
  }
  return <button onClick={handleClick}>Place order</button>;
}
```

The key is cleared automatically once the run reaches a terminal state.

> **When to opt in.** Most calls do not need `scopeKey`. The default auto-id survives connection drops and worldant restarts. Add `scopeKey` only when users might navigate away and back while waiting for a long-running result.

| Option | Scope | Persists across reload? | Use when |
|---|---|---|---|
| (none) | per-call, in-memory | No | Default — handles all transient failures |
| `runId` | caller-controlled | Only if you store it | You own id lifecycle (server actions, tests) |
| `scopeKey` | named slot, localStorage | Yes | User may navigate away mid-run |
