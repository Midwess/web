# How it works

PgPaw is a single Rust binary that reads from an embedded [pglite-rs](/pglite-rs) logical replica and serves three things over HTTP: immutable CDN-cacheable snapshots, live SSE deltas, and a health check. The cache key is content-addressed (`sql-fingerprint + version`) and the version is derived from the replica's LSN watermark — so a relevant upstream write is the only thing that ever re-issues a URL.

This page walks the architecture top-down, then drills into the cache invalidation flow, the SSE stream, and the RLS seam.

## Architecture

Upstream Postgres runs the `wal_level=logical` setup, the publication, and the DDL event trigger. The pglite-rs replica mirrors it through `pgoutput` logical replication. PgPaw sits in front of the replica and serves HTTP.

What each piece does:

- **ReadClassifier** parses the SQL with `sqlparser`, rejects anything that is not a single read-only `SELECT`, and extracts the table list plus equality filters (anchors). See [SQL classifier](/pgpaw/sql-classifier).
- **VersionIndex** consumes CDC transactions from the pglite-rs replica and tracks the highest LSN per `(table, column, value)` for anchored queries, and per-table for `TRUNCATE` and unanchored queries. The version of a query is the max LSN across the tables it touches.
- **QueryCache** (moka) stores `key = sql-fingerprint:version → result`. The fingerprint is stable for the parsed statement, the version is monotonic with upstream writes — so the cache key is immutable until a relevant write actually occurs.
- **CdcBridge** is the thread that bridges pglite-rs's change broadcast to the VersionIndex. Each committed upstream transaction bumps the LSN entries for the tables (and rows) it touched.
- **LiveHub** re-runs the user SQL after each committed transaction that affects a touched table, diffs the new rows against the previous snapshot, and pushes row-level deltas to all subscribers of `?live=true`.
- **JWT verify + RLS seam** verifies the bearer token, sets the token's role via `SET LOCAL ROLE` inside a pinned transaction, and runs the query as that role. The replica's own RLS policies filter the rows. See [Authorization](/pgpaw/authorization).

The five properties of this flow that matter:

- **The URL is the content.** Two clients that fetch `/q/9a4f/v=42` always get the same bytes, even years later, even if every other row in the table has changed.
- **Only relevant writes invalidate.** A `WHERE id = 7` query is anchored on `(users, id, 7)`. An `UPDATE` to row 12 does not bump the LSN for `7`, so the URL stays valid. Joins and queries without equality filters fall back to per-table versioning.
- **The CDN never re-validates.** `Cache-Control: public, max-age=31536000, immutable` is sent on every snapshot response. The edge can hold the body until the URL itself changes.
- **Watermark, not wall-clock.** The version is the LSN of the most recent relevant commit, not a timestamp. Two replicas see the same version for the same commit order. Cache invalidation is deterministic.
- **The `303 See Other` redirect keeps `POST` semantics.** A `POST` body is non-cacheable, but the `Location` header on a `303` is cacheable. Browsers, fetch, and `curl -L` all follow the redirect to the immutable URL.

The split is **fail-closed**. Anything that cannot be proven public is treated as access-controlled. Concretely:

- **Public** — every referenced table has RLS disabled **and** `SELECT` is granted to `PUBLIC`. The query is served from the shared cache with `Cache-Control: public, max-age=259200, immutable` (the version URL stays valid for a year after the version is issued). No token required.
- **Access-controlled** — anything else (RLS enabled, or not `PUBLIC`-readable). Requires a valid token, executes under the token's role, returns rows inline with `Cache-Control: private, no-store`. Never cached, never CDN-served. The `?live=true` variant streams deltas computed under the token's role.

The RLS seam uses `SET LOCAL ROLE` inside a pinned `Transaction` on the pglite-rs replica. The replica's own RLS policies filter the rows. PgPaw only verifies tokens; it never issues them.

The stream stays open and emits another batch on the next relevant commit. If the replication feed lags and transactions are dropped, PgPaw emits a `reset` control event so clients re-load the snapshot instead of diverging.

## Why the replica is the right place to run reads

PgPaw does not parse, plan, or evaluate SQL itself. Every read is `SELECT …` against the pglite-rs replica — the same embedded Postgres engine, with the same types, the same `EXPLAIN` output, the same RLS semantics. The replica is the working set; PgPaw is the cache layer in front of it. That is what makes per-row anchoring, RLS under the token's role, and joins work without a custom evaluator.

The replica is also how PgPaw stays in sync with the upstream: DDL is captured by an event trigger that calls `pg_logical_emit_message('pglite_ddl', '')`, the replica re-introspects and applies the delta, and the new shape is what the classifier sees on the next query. No restarts, no manual schema sync.

## What it costs

- **One upstream connection** for the logical replication client. SCRAM auth, `START_REPLICATION`, pgoutput decode — same path as pglite-rs's `Replica`.
- **One upstream connection** for the optional DDL signal trigger (installed by `pgpaw init`).
- **In-process memory** for the `moka` cache. Default 256 MiB. Tune via `--cache-size-bytes`.
- **Replica disk** for the embedded Postgres data dir. Same sizing rules as pglite-rs — about 23 MB for the engine, plus your data.
- **No external state.** No Redis, no Memcached, no separate schema service.

For typical web traffic, the cache hit ratio dominates everything else — most reads return the same `(sql-fingerprint, version)` result over and over. The cost is bounded by the upstream's CDC rate, not the read rate.
