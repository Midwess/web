# How remote functions work

When `net` is enabled on a world, worldant exposes two HTTP surfaces. The npm registry side answers `npm install`; the runs side answers every call the generated client makes.

## Contract extraction

At build time (`withWorldant` in Next.js, or `gen-contract` for standalone setups), worldant parses your TypeScript source files using the TypeScript compiler API. It finds every exported function annotated with `"use workflow"` or `"use step"` and reads its parameter and return types directly from the AST — no runtime reflection, no decorators.

```ts
export async function placeOrder(order: Order): Promise<OrderReceipt> {
  'use workflow';
  // ...
}
```

The extracted signatures are written to `worldant.catalog.json`. The catalog feeds both the code-generated client and the npm package the world serves at `/__worldant/v1/npm`.

## The call path

When a consumer calls a generated wrapper like `placeOrder(order)`, `worldant-client` drives a three-stage sequence:

1. `POST /__worldant/v1/runs?wait=N` — submit the run with a `runId` (auto-generated or supplied) and wait up to `N` seconds for inline completion. If the workflow finishes within that window, the response carries the result and the call resolves immediately — no second request needed.
2. `GET /__worldant/v1/runs/:id` — if the POST times out without a terminal result, the client polls this endpoint every `pollMs` milliseconds until the run completes.
3. Network or `5xx` errors on either step are retried with exponential backoff against **the same `runId`** — the run is never submitted twice.

## Run IDs and idempotency

Every call is idempotent by default. `worldant-client` auto-generates a conformant `wrun_<ULID>` for each call, in memory, scoped to that call. If the network drops or worldant restarts mid-flight, retrying resumes the **same run** — never a duplicate.

```ts
// Auto-idempotent — no setup required
const receipt = await placeOrder(order);
```

The `runId` is valid for any world backend (SQLite, Postgres, or Vercel) because it is a real `wrun_<ULID>` — the same format those worlds expect.

The client generates the id; the world deduplicates on its run primary key. There is no server-side idempotency table.

### Explicit run ID

Pass your own id to control dedup from outside the client:

```ts
import { generateRunId } from 'worldant-client';

const runId = generateRunId(); // wrun_<ULID>
const receipt = await placeOrder(order, { runId });
// Any retry with the same runId resolves the same run
```

`generateRunId()` is exported from `worldant-client` and from the generated `worldant/client.ts` module.

### `scopeKey` — survive a page reload (opt-in)

The auto-id is in-memory. To survive a full page reload (the user navigates away and back during a long-running order), persist the id with a `scopeKey`:

```ts
const receipt = await placeOrder(order, { scopeKey: `order:${cartId}` });
```

The client mints a run id, stores it under `worldant:run:order:<cartId>` in `localStorage` (browser) or an in-memory map (SSR / Node), and on the next call with the same key it resumes the stored id. The key is cleared once the run is terminal.

> **When to opt in.** Most calls do not need `scopeKey` — the auto-id handles connection drops and worldant restarts without any setup. Use `scopeKey` only when the user might leave and return to a page that should pick up a run that is still in progress.

## Codec — `Uint8Array` and `Date`

Plain JSON cannot round-trip `Uint8Array` or `Date`. Worldant's codec handles both:

| Type | Encoding |
|---|---|
| `Uint8Array` | Base64-encoded string, decoded back to `Uint8Array` on the client |
| `Date` | ISO 8601 string, reconstructed as `Date` on the client |
| Everything else | Standard JSON |

Codec runs on both sides — no manual serialization in workflow code or consumer code.

```ts
export async function processFile(data: Uint8Array): Promise<{ processedAt: Date }> {
  'use workflow';
  // data arrives as Uint8Array in the workflow body
  // processedAt arrives as Date in the caller
  return { processedAt: new Date() };
}
```

## Security

`defaultConfiguration` (world id, base URL, poll interval) is embedded in the generated package and is safe to expose publicly — it never contains a token. Supply auth at runtime:

```ts
import { configure } from 'orders';
configure({ token: process.env.NEXT_PUBLIC_WORLD_TOKEN });
```

The token is sent as `Authorization: Bearer <token>` on every request. See [Security](/worldant/remote-functions#security) in the overview for the full model.
