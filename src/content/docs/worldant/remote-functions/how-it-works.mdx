# How remote functions work

When `net` is enabled on a world, worldant exposes two HTTP surfaces. The npm registry side answers `npm install`; the runs side answers every call the generated client makes.

## Contract extraction

At build time (`withWorldant` in Next.js, or `gen-contract` for standalone setups), worldant parses your TypeScript source files using the TypeScript compiler API. It finds every exported function annotated with `"use workflow"` or `"use step"` and reads its parameter and return types directly from the AST — no runtime reflection, no decorators.

```ts
export async function placeOrder(order: Order): Promise<OrderReceipt> {
  'use workflow';
  // ...
}
```

The extracted signatures are written to `worldant.catalog.json`. The catalog feeds both the code-generated client and the npm package the world serves at `/__worldant/v1/npm`.

## The call path

When a consumer calls a generated wrapper like `placeOrder(order)`, `worldant-client` drives a standard HTTP sequence:

1. `POST /__worldant/v1/runs` — start the run, get a run id back.
2. `GET /__worldant/v1/runs/:id/stream` — SSE stream of step events and the final `returnValue`.
3. If SSE is unavailable (non-browser, firewall, etc.) — poll `GET /__worldant/v1/runs/:id` every `pollMs` milliseconds.

## Codec — `Uint8Array` and `Date`

Plain JSON cannot round-trip `Uint8Array` or `Date`. Worldant's codec handles both:

| Type | Encoding |
|---|---|
| `Uint8Array` | Base64-encoded string, decoded back to `Uint8Array` on the client |
| `Date` | ISO 8601 string, reconstructed as `Date` on the client |
| Everything else | Standard JSON |

Codec runs on both sides — no manual serialization in workflow code or consumer code.

```ts
export async function processFile(data: Uint8Array): Promise<{ processedAt: Date }> {
  'use workflow';
  // data arrives as Uint8Array in the workflow body
  // processedAt arrives as Date in the caller
  return { processedAt: new Date() };
}
```

## Security

`defaultConfiguration` (world id, base URL, poll interval) is embedded in the generated package and is safe to expose publicly — it never contains a token. Supply auth at runtime:

```ts
import { configure } from 'orders';
configure({ token: process.env.NEXT_PUBLIC_WORLD_TOKEN });
```

The token is sent as `Authorization: Bearer <token>` on every request. See [Security](/worldant/remote-functions#security) in the overview for the full model.
