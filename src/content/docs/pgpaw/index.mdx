# PgPaw

PgPaw is a single Rust binary that sits in front of a Postgres and serves **CDN-cacheable, version-addressed query snapshots** *and* **live realtime updates** from the same endpoint. Unlike shape- or ZQL-based sync engines, you query with plain Postgres SQL — no bespoke query language to learn.

It embeds a [pglite-rs](/pglite-rs) instance as a logical replica of your upstream Postgres, runs every read inside that replica, and gives you three things that are usually painful to bolt on:

1. **Precise, watermark-derived cache invalidation** keyed on per-table / per-PK LSN state.
2. **CDN-cacheable, version-addressed snapshots** — a `303` redirect to a content-hash URL (`/q/{hash}/{version}`). The redirect itself carries `Cache-Control: no-store`; the snapshot resource it points at is served with `Cache-Control: public, max-age=259200` (72h) plus an `ETag`.
3. **Realtime deltas over SSE** — first event is a snapshot (a `url` pointer for public queries, inline `rows` for access-controlled ones), then `insert` / `update` / `delete` events per affected row, each carrying a `txid`.

## Features

- **Single static binary.** `cargo install pgpaw` or `npm install -g pgpaw`. One process, one config, one upstream.
- **Plain Postgres SQL.** `SELECT id, email FROM users WHERE id = 7` — no DSL, no codegen, no schema sync. Drop it into [TanStack Query](https://tanstack.com) / DB, REST clients, anything that speaks HTTP.
- **Version-addressed, CDN-cacheable responses.** Every snapshot lives at `/q/{hash}/{version}`, served with `Cache-Control: public, max-age=259200` (72h) plus an `ETag` equal to the cache key `{hash}:{version}`. The URL is the content+version: a data change yields a new URL rather than mutating an existing one.
- **Per-row cache anchoring.** A `WHERE pk = literal` becomes a per-row anchor — writes to other rows do not invalidate the cache entry. Per-table fallback for joins and unanchored queries.
- **Live SSE deltas.** `POST /query?live=true` opens a stream: a snapshot event, then row-level `insert` / `update` / `delete` events for affected rows (each carrying a `txid`), then a terminal `up-to-date {txid}` for that recompute round — emitted even when the diff is empty, so a client awaiting a `txid` cannot hang.
- **Native Postgres Row-Level Security.** Access-controlled queries are verified by JWT, then run as the token's role inside the embedded replica. Your existing RLS policies are the authority — PgPaw only verifies tokens, never issues them.
- **Public vs access-controlled split.** Public data (RLS disabled, `GRANT SELECT TO PUBLIC`) serves from the shared cache. Anything else is private, per-token, never cached.
- **Logical replication out of the box.** One `pgpaw init` sets `wal_level=logical`, creates the publication, and installs a DDL event trigger (`pglite_ddl_watch`, firing on `ddl_command_end`). After that, the replica stays in sync automatically — DDL, schema changes, and all.
- **Read-only by construction.** The SQL classifier rejects `INSERT` / `UPDATE` / `DELETE` / `MERGE` / DDL, `FOR UPDATE` / `FOR SHARE`, `SELECT INTO`, multi-statement input, side-effecting functions (`nextval` / `setval` / `pg_advisory_*`), volatile-read functions (`now` / `random` / `clock_timestamp` / `gen_random_uuid` / …), and references to non-replicated tables. The replica is also read-only to PgPaw.
- **Built on [pglite-rs](/pglite-rs).** The replica is the same embedded engine that powers the rest of the stack. Logical replication, DDL tracking, security catalog mirroring, change broadcast — all inherited.
- **Sub-millisecond cache hits.** `moka` in-process cache. The key is the string `{hash}:{version}`, where `hash` is `{fingerprint:x}` (a `DefaultHasher` of the normalized statement) and `version` is the LSN watermark from `VersionIndex`; that same string doubles as the `ETag`. The key is stable until a relevant write actually occurs.
- **Health endpoint.** `GET /healthz` reports replica watermark and `ok` / `halted` status.

## What it is not

- **Not a write proxy.** PgPaw is read-only. Writes go to upstream through your existing path.
- **Not a generic SQL cache.** It does not serve OLTP workloads — every read must fit the classifier and the publication.
- **Not a Postgres connection pooler.** The replica is a single embedded engine; the upstream connection is one logical-replication client.
- **Not a sync engine DSL.** It speaks plain Postgres SQL, not a custom query language. If you want a sync engine with shape subscriptions, look at [ElectricSQL](https://electric-sql.com) or [Rocicorp Zero](https://zero.rocicorp.dev) directly.
