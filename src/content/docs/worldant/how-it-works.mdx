# How it works

Worldant is a self-hosted, durable `@workflow/world` implementation on embedded SQLite. It ships as two fate-shared units: the in-app library (`@midwess/worldant` — a TS adapter over a native addon backed by `world-core`) and the `worldant` supervisor process that sits in front of it. Three things to understand: the layers that make it up, how a workflow run flows through them, and how the supervisor keeps the process alive.

## Layers

The system has two OS-level processes: your Node app, and the `worldant` CLI that sits in front of it. The Node app hosts the workflow runtime and the storage engine in one process.

| Layer | Where | Role |
|-------|-------|------|
| App code | `your files` | Workflows, steps, hooks, streams. Imports `workflow` and `workflow/api` |
| `@workflow/core` | npm | Compiler transform (`"use workflow"` / `"use step"`), runtime replay, determinism guard |
| `@midwess/worldant` | `src/` | TS shim that implements the `@workflow/world` interface on top of the native `World` |
| `World` (napi addon) | `crates/world-napi` | Exposes `world-core` to Node as a `cdylib`, marshalling every method as JSON strings |
| `world-core` (Rust) | `crates/world-core` | Reduce-to-SQL event log, queue, and streams over embedded SQLite. The durable layer |
| Supervisor | `src/supervisor.ts` | Hijacks `http.Server.prototype.listen`, reports the private socket up to the CLI |
| `worldant` CLI | `crates/worldant-cli` | Owns the public socket, forwards traffic, decides when the process can stop |

`tokio` only appears at the napi and CLI edges. `world-core` and `world-config` carry no async runtime — their `async fn` signatures have synchronous `rusqlite`/`r2d2` bodies, so the durable layer is runtime-agnostic.

## A run, start to finish

Six steps from trigger to terminal. The world writes events to SQLite as the workflow progresses; every step and event write is one transaction.

<DChart label="Run lifecycle" title="Six steps from trigger to terminal, each committing before the next runs">

    <DBox badge={1} title="trigger" sub={["events.create", "run_created"]} />
    <DArrow dir="right" kind="info" />

    <DArrow dir="right" kind="info" />
    <DBox kind="accent" badge={3} title="replay" sub={["workflow body", "to next await"]} />
    <DArrow dir="right" kind="info" />
    <DBox badge={4} title="commit" sub={["event + queue", "in 1 tx (CBOR)"]} />
    <DArrow dir="right" kind="info" />
    <DBox badge={5} title="continue" sub={["next step /", "hook / wait"]} />
    <DArrow dir="right" kind="info" />
    <DBox kind="success" badge={6} title="terminal" sub={["run_completed", "event written"]} />

1. **Trigger.** Your code calls `start(checkout, [order])` from `workflow/api`. The TS shim calls `world.events.create(...)` on the native `World`, which inserts a `run_created` event into SQLite and enqueues a workflow queue message (prefix `__wkf_workflow_`).
2. **Dispatch.** The in-process queue worker calls `claim_next`, picks up the message, replays the workflow body to its first `await`, and reaches the first step call.
3. **Step commit.** The step function is a regular JS function in the steps bundle. Its return value (or thrown error) is wrapped as a step event, committed to SQLite atomically with the queue message, and returned to the workflow body.
4. **Continue.** The workflow body resumes with the step's return value. It calls the next step, sleeps, awaits a hook, or returns.
5. **Hooks and streams.** `createHook()` creates a hook record and a wait the workflow is blocked on. External code calls `world.hooks.getByToken()` and resolves the hook — the workflow body wakes up. Streams are similar but for append-only byte chunks.
6. **Terminal.** The workflow body returns or throws. The world writes a `run_completed` (or failed) event and stops processing the run.

### Inside an event create: reduce to SQL

Every state change is an event, and every event reduces itself into SQL. There is no ORM and no separate event-sourcing framework — the reducer *is* the write path.

`events_create` resolves the run id (a `run_created` event with no id mints a `wrun_<ULID>`), rejects any `spec_version` greater than 3 (`RunNotSupported`), and builds an `EventCommand`. The `ambassador`-generated `Reduce` enum dispatches each event to its per-event reducer (run, step, hook, wait), which `UPDATE`s/`INSERT`s the `world_*` tables and `append_event()`s into `world_events`. JSON payloads are CBOR-encoded into blob columns; reads project them back to JSON through the custom `cbor_json()` SQLite scalar function.

### How a queued job is dispatched and retried

The queue is not a broker or a second database — it is the `world_jobs` SQLite table, drained by an in-process worker.

<DChart label="Queue" title="The in-process worker claims a job, dispatches it, and maps the reply to Done, Retry, or Failed">
  <DRow gap={12} align="center" wrap>
    <DBox title="QueueWorker" sub={["Semaphore · concurrency 128", "tokio (napi edge)"]} />
    <DArrow dir="right" kind="info" />
    <DBox title="claim_next" sub={["atomic UPDATE · attempts+1", "15s lease reclaims abandoned"]} />
    <DArrow dir="right" kind="info" />
    <DBox title="NapiDispatch → JS dispatcher" sub={["ThreadsafeFunction · prefix → handler", "__wkf_step_ = step · __wkf_workflow_ = flow", "ActivityGuard ++active (supervisor busy)"]} />
  </DRow>
  <DNote center>DispatchOutcome</DNote>
  <DArrow dir="down" kind="info" />
  <DRow gap={14} align="start">
    <DBox grow={1} kind="success" title="Done" sub={["complete_job", "(delete the row)"]} />
    <DBox grow={1} kind="warn" title={"Retry { delay }"} sub={["fail_job(delay) — handler", "asked to wait, then re-run"]} />
    <DCol>
      <DBox kind="error" title="Failed" sub={["exp-backoff fail_job", "base 1s → cap 300s"]} />
      <DArrow dir="down" kind="error" dashed />
      <DBox kind="error" title="dead-letter after 25 attempts" sub="attempts ≥ max_attempts → stop, log and drop" />
    </DCol>
  </DRow>

A workflow or step enqueues via `world.queue(name, msg, opts)`, which inserts into `world_jobs` with `kind` derived from the queue-name prefix (`__wkf_step_` = step, `__wkf_workflow_` = flow). The worker loops: it takes a `Semaphore` permit, calls `claim_next` (an atomic `UPDATE ... RETURNING` that bumps `attempts` and reclaims any job whose 15-second lease has expired), then dispatches through `NapiDispatch` to the JS dispatcher `ThreadsafeFunction`. The dispatcher matches the prefix to a registered handler, runs it, and the reply maps to a `DispatchOutcome`: `Done` deletes the job (`complete_job`), `Retry { delay }` reschedules it, and `Failed` applies exponential backoff (base 1s, capped at 300s) or dead-letters after 25 attempts. While a handler runs, an `ActivityGuard` increments the control server's active counter so the supervisor sees the app as busy.

## Crash recovery

When a process restarts with the same `dataDir`, the world reads the in-flight runs (`reenqueue_active_runs`) and replays them from the durable log. A step whose `StepStatus` is already terminal (`completed`) is **memoized** — its function body never runs again, and the `*_created` events that recorded it dedup through a partial `UNIQUE` index (`ON CONFLICT`) so replays never double-write. A step that crashed mid-body is re-enqueued for retry.

A retry re-executes the step function with the same arguments, but with `getStepMetadata().attempt` incremented. The step can branch on `attempt` to throw `RetryableError` on the first try and succeed on the second.

The log is the source of truth. The TS runtime is a deterministic interpreter that re-derives the same control flow from the log on every wake-up.

## The supervisor

The supervisor is the scale-to-zero half of the system. It is a separate process — the `worldant` CLI — that sits in front of your app and decides when your app process needs to be running.

```bash
worldant start -- node dist/main.js
```

What happens:

1. The CLI starts the child process with `WORLDANT_CONTROL_FD` set, pointing at an announce pipe.
2. The child process imports `@midwess/worldant`. The `connectSupervisor()` call sees the FD and patches `http.Server.prototype.listen`.
3. When the app calls `server.listen(3000)`, the patched `listen` actually binds a **private Unix socket** (at `$XDG_RUNTIME_DIR/worldant-<hash>.<port>.app.sock`), and reports the original (host, port, privateSock) tuple up the announce pipe.
4. The library then opens a control Unix socket and streams `Register` / `Listen` / `State` / `Wake` frames. The CLI records the listen into its `SharedState`, binds the public TCP port (host taken from the reported listen, falling back to `0.0.0.0`), accepts incoming connections, and forwards every byte to the private socket with `copy_bidirectional`.

5. When the CLI sees no `State(active)` work and no open connections for `graceMs` (after at least `minAliveMs` of warmup), it `SIGTERM`s the child and clears the app sockets — but **keeps the public TCP listener bound**. The app also reports a `Wake(at)` frame carrying its next durable timer (`next_wake`), so the supervisor knows when to wake it even with no inbound traffic.
6. The next inbound connection — or the `Wake` timer firing — respawns the child. It reopens the same `dataDir`, reenqueues active runs, and resumes from the log. Connections held during the wake (up to `wakeHoldMs`) are pushed to the woken app.

This is what makes the system cheap to leave running. An idle workflow app can sit at zero CPU between requests and pick up exactly where it left off. The supervisor never opens the World or the database — durability lives entirely in the app library.

## Storage

A single SQLite database per app, configured by `dataDir`. WAL mode, `busy_timeout` set to 5000ms, a single writer with an `r2d2` reader pool (`max_size` 8). `rusqlite` is pinned to `0.37` with the `bundled` feature, so SQLite is statically linked into the native addon that loads inside the Node process — there is no external SQLite dependency and no compiler invocation at install time.

The data dir is opened once on boot. **Two processes must not open the same dir concurrently.** The code-first `id` (derived from the package name plus a `sha256` of `name:root`) keys both the library's dedup lock and the supervisor's `flock`-based `IdLock`, so exactly one opener wins per `id`. If you want multiple replicas, give each a different `id` so the lock and socket don't collide.
