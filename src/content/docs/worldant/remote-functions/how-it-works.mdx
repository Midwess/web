---
title: How it works
description: Request flow for Worldant remote function calls.
---

The daemon serves a stable protocol:

```text
GET  /__worldant/v1/registry
GET  /__worldant/v1/registry.d.ts
GET  /__worldant/v1/contract/:etag.tgz
GET  /__worldant/v1/npm/:package

POST /__worldant/v1/calls
GET  /__worldant/v1/calls/:callId
GET  /__worldant/v1/calls/:callId/stream

POST /__worldant/v1/runs
GET  /__worldant/v1/runs/:runId
GET  /__worldant/v1/runs/:runId/stream

POST /__worldant/v1/subscriptions
GET  /__worldant/v1/subscriptions/:subId/stream
```

The registry and contract endpoints describe the active `.dist` snapshot. The
call and run endpoints execute snapshot-backed artifacts. Subscription streams
deliver authorized SQL-backed updates.

Idempotency is part of the protocol:

```text
mutation call + same key + same payload -> existing result
mutation call + same key + different payload -> conflict
workflow run + same run id -> existing run
event subscription + cursor -> resume from cursor
live query subscription -> reconnect by re-running latest query
```
