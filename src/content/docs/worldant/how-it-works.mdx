# How it works

Worldant is a single-process implementation of the Vercel Workflow world. Three things to understand: the layers that make it up, how a workflow run flows through them, and how the supervisor keeps the process alive.

## Layers

The system has two OS-level processes: your Node app, and the `worldant` CLI that sits in front of it. The Node app hosts the workflow runtime and the storage engine in one process.

| Layer | Where | Role |
|-------|-------|------|
| App code | `your files` | Workflows, steps, hooks, streams. Imports `workflow` and `workflow/api` |
| `@workflow/core` | npm | Compiler transform (`"use workflow"` / `"use step"`), runtime replay, determinism guard |
| `@midwess/worldant` | `src/` | TS adapter that implements the `@workflow/world` interface on top of the native `World` |
| `World` (NAPI) | `crates/world-napi` | Exposes the Rust storage engine to Node as a `cdylib` |
| `world-core` (Rust) | `crates/world-core` | Storage + Queue + Streamer over embedded SQLite. The durable layer |
| Supervisor | `src/supervisor.ts` | Hijacks `http.Server.prototype.listen`, reports the private socket up to the CLI |
| `worldant` CLI | `crates/worldant-cli` | Owns the public socket, forwards traffic, decides when the process can stop |

`tokio` only appears at the NAPI and CLI edges. `world-core` and `world-config` carry no async runtime — the durable layer is runtime-agnostic.

## A run, start to finish

Six steps from trigger to terminal. The world writes events to SQLite as the workflow progresses; every step and event write is one transaction.

1. **Trigger.** Your code calls `start(checkout, [order])` from `workflow/api`. The TS adapter calls `world.events.create(...)` on the native `World`, which inserts a `run_created` event into SQLite and enqueues a `workflow` queue message.
2. **Dispatch.** The world's queue loop picks up the message, replays the workflow body to its first `await`, and reaches the first step call.
3. **Step commit.** The step function is a regular JS function in the steps bundle. Its return value (or thrown error) is wrapped as a step event, committed to SQLite atomically with the queue message, and returned to the workflow body.
4. **Continue.** The workflow body resumes with the step's return value. It calls the next step, sleeps, awaits a hook, or returns.
5. **Hooks and streams.** `createHook()` creates a hook record and a queue message the workflow is waiting on. External code calls `world.hooks.getByToken()` and resolves the hook — the workflow body wakes up. `getWritable()` is similar but for byte streams.
6. **Terminal.** The workflow body returns or throws. The world writes a `run_completed` (or failed) event and stops processing the run.

## Crash recovery

When a process restarts with the same `dataDir`, the world reads the in-flight runs and replays them from the durable log. A step whose `completed` event was already committed is **memoized** — its function body never runs again. A step that crashed mid-body is re-enqueued for retry.

A retry re-executes the step function with the same arguments, but with `getStepMetadata().attempt` incremented. The step can branch on `attempt` to throw `RetryableError` on the first try and succeed on the second.

The log is the source of truth. The TS runtime is a deterministic interpreter that re-derives the same control flow from the log on every wake-up.

## The supervisor

The supervisor is the scale-to-zero half of the system. It is a separate process — the `worldant` CLI — that sits in front of your app and decides when your app process needs to be running.

```bash
worldant start -- node dist/main.js
```

What happens:

1. The CLI starts the child process with `WORLDANT_CONTROL_FD=3` set, pointing at an announce pipe.
2. The child process imports `@midwess/worldant`. The `connectSupervisor()` call sees the FD and patches `http.Server.prototype.listen`.
3. When the app calls `server.listen(3000)`, the patched `listen` actually binds a **private Unix socket** (at `$XDG_RUNTIME_DIR/worldant-<hash>.<port>.app.sock`), and reports the original (host, port, privateSock) tuple up the announce pipe.
4. The CLI binds the public port, accepts incoming connections, and forwards bytes to the private socket.

5. When the CLI sees no traffic for `graceMs` (and at least `minAliveMs` has passed since the last request), it kills the child. The private socket goes with it.
6. The next request to the public port restarts the child from scratch. SQLite replays the log. The workflow resumes.

This is what makes the system cheap to leave running. An idle workflow app can sit at zero CPU between requests and pick up exactly where it left off.

## Storage

A single SQLite database per app, configured by `dataDir`. WAL mode, `busy_timeout` set, single writer with a `r2d2` reader pool. `rusqlite` is pinned to `0.37` with the `bundled` feature — the binary ships with SQLite statically linked and matches `deno_runtime`'s transitive `libsqlite3-sys` so there is exactly one `sqlite3` in the process.

The data dir is opened once on boot. **Two processes must not open the same dir concurrently** — the supervisor uses a dedup lock to prevent that. If you want multiple replicas, give each a different `id` so the lock and socket don't collide.
