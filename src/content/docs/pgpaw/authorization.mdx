title: Authorization
description: Configure JWT verification and Postgres RLS enforcement.
---

PgPaw relies on Postgres for row-level security. JWT verification only decides which role and claims are set before running a private query.

## Configure verification

Use exactly one key source:

```bash
pgpaw serve \
  --jwt-secret "$JWT_SECRET" \
  --jwt-role-claim role
```

or:

```bash
pgpaw serve \
  --jwt-public-key "$JWT_PUBLIC_KEY" \
  --jwt-role-claim role
```

`--jwt-jwks-url` exists as a flag, but the cloned code returns a configuration error because JWKS verification is not implemented yet.

## Supported algorithms

- `--jwt-secret`: HS256.
- `--jwt-public-key`: RS256 when the PEM parses as RSA, otherwise ES256 when it parses as EC.

Audience validation is disabled. Expiration is still validated by `jsonwebtoken`.

## Request header

```http
Authorization: Bearer <token>
```

The role claim must be a string. If `--jwt-role-claim org_role` is set, the token must include:

```json
{ "org_role": "member" }
```

## Privacy verdict

A query is private if any touched table has RLS enabled or lacks `PUBLIC SELECT`. Public queries ignore JWT and use the shared snapshot path. Private queries require JWT and are never cached publicly.

Database denial SQLSTATEs `42501`, `42704`, and `28000` map to `403`.
