# SQL classifier

PgPaw serves read-only Postgres SQL against the replicated tables. The classifier parses the input with `sqlparser` and rejects anything that is not a single read-only `SELECT` against tables present in the publication. The classification also extracts the table list and equality filters (anchors) that drive per-row cache versioning.

## What's allowed

A query is served if **all** of these hold:

1. It is a single statement. No `select 1; select 2`.
2. The top-level statement is `SELECT`. Not `INSERT`, `UPDATE`, `DELETE`, `MERGE`, `WITH ... DELETE`, `WITH ... UPDATE`, `VALUES`, `TABLE`, `EXPLAIN`, or DDL.
3. The `SELECT` does not lock rows. Not `FOR UPDATE`, `FOR NO KEY UPDATE`, `FOR SHARE`, `FOR KEY SHARE`.
4. The `SELECT` does not write a variable. Not `SELECT ... INTO`.
5. It does not call a side-effecting or non-deterministic function from the reject list (see below).
6. Every referenced table is present in the publication. Tables the publication does not cover cannot be queried.

If all of these hold, the query is parsed to extract the table list and equality filters, fingerprinted, versioned against the LSN, and served.

## What's rejected

The classifier rejects (returns `400 ParseError`):

- **Multi-statement input** — `select 1; select 2`. The body is parsed as a single statement; anything else is a reject.
- **Writes** — `INSERT`, `UPDATE`, `DELETE`, `MERGE`, `TRUNCATE`.
- **DDL** — `CREATE`, `DROP`, `ALTER`, `COMMENT`, `GRANT`, `REVOKE`. DDL is only allowed inside the upstream, never as a query against PgPaw.
- **Locking reads** — `SELECT ... FOR UPDATE`, `SELECT ... FOR SHARE`, `SELECT ... FOR NO KEY UPDATE`, `SELECT ... FOR KEY SHARE`.
- **Variable writes** — `SELECT col INTO var FROM ...` (PL/pgSQL idiom; not a thing in pglite-rs anyway, but rejected up front).
- **Side-effecting functions** — `nextval`, `setval`, `pg_advisory_lock`, `pg_advisory_xact_lock`, `pg_advisory_lock_shared`, `pg_advisory_xact_lock_shared`, `pg_advisory_lock_try`, `pg_advisory_xact_lock_try`, `pg_try_advisory_lock`, `pg_advisory_unlock`, `pg_advisory_unlock_all`, `pg_advisory_unlock_shared`, `pg_advisory_unlock_all_shared`.
- **Non-deterministic reads** — `now`, `current_timestamp`, `current_time`, `localtimestamp`, `localtime`, `statement_timestamp`, `clock_timestamp`, `timeofday`, `transaction_timestamp`, `gen_random_uuid`, `uuid_generate_v4`, `random`, `random_normal`. (The list matches the [pglite-rs non-deterministic funcs list](/pglite-rs/configuration#pgliteoptions) — it is the same set the replica considers unsafe for cache keys.)
- **References to non-replicated tables** — `select * from secrets` when `secrets` is not in the publication. The error message names the offending table.

A rejected query returns `400` with a `ParseError` envelope:

```json
{ "name": "ParseError", "message": "table `secrets` is not available in this cache (not replicated)" }
```

## Per-row cache anchoring

A `WHERE col = literal` on a single table that is either the table's primary key or a column on a `REPLICA IDENTITY FULL` table becomes a per-row anchor. The `VersionIndex` keeps a separate LSN per `(table, column, value)`. Writes to other rows do not bump the anchor's LSN, so the cache entry stays valid.

```sql
-- Anchored: (orders, id, 7)
select id, total from orders where id = 7;

-- Anchored on REPLICA IDENTITY FULL table: (orders_log, id, 99)
select * from orders_log where id = 99;

-- NOT anchored: no equality filter on PK or REPLICA IDENTITY FULL column
select * from orders where status = 'paid';

-- Joins: per-table fallback (see below)
select o.id, c.email
  from orders o
  join customers c on c.id = o.customer_id
 where o.id = 7;
```

When the query has no anchor — no equality filter on a PK or `REPLICA IDENTITY FULL` column — the version falls back to the per-table LSN. Any write to the table bumps the version. Joins also fall back to per-table versioning for every table in the join.

The cache key is `(sql-fingerprint, version)`. Per-row anchoring means two queries on the same table with different row filters never invalidate each other.

## What "anchorable" means

A column is anchorable if either:

- it is the table's primary key, OR
- the table has `REPLICA IDENTITY FULL` set (every column is part of the replica identity), AND a `USING` or `ON` clause references exactly that column with a literal (no `IN`, no subquery, no function call)

`pgpaw init` does not change `REPLICA IDENTITY` — that is the operator's choice on the upstream. Tables without a PK and without `REPLICA IDENTITY FULL` get per-table versioning only.

## Functions in the allow list

The classifier only rejects the explicit deny list above. Every other function passes through to the replica. The replica enforces the rest — the same rules as a normal pglite-rs query.

`pg_catalog` lookups (`current_database`, `current_user`, `current_schema`, `pg_get_userbyid`, `version()`) and most read-only functions (`count`, `sum`, `avg`, `array_agg`, `string_agg`, `jsonb_agg`, `json_agg`, `coalesce`, `nullif`, `lower`, `upper`, `length`, `substring`, `replace`, `to_char`, `date_trunc`, `date_part`, `extract`, `generate_series` with bounded input) are fine.

If a function is not in the deny list and not marked `VOLATILE` in `pg_proc`, it passes. The classifier errs on the side of "let the replica decide" — the replica's permission system is the authority.

## Working with the classifier

- If a query is rejected unexpectedly, the error message names the offending construct. The most common cause is a forgotten `in publication` for a referenced table.
- If a query is served but always returns a per-table version (and the cache invalidates on every write), check that the equality filter is on a PK or `REPLICA IDENTITY FULL` column, and that the literal is a constant the classifier can extract.
- `EXPLAIN` is not supported — PgPaw does not expose the replica's planner to clients. Use a `psql` connection to the upstream if you need plan inspection.
