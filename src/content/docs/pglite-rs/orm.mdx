# ORM integration

`PGlite::unix_uri()` returns a `postgresql://user@localhost/db?host=…` DSN pointed at a private unix socket — the same string every Postgres driver accepts. Pass it to SQLx, SeaORM, Diesel, or `tokio-postgres` exactly as you would for a remote Postgres. The mode (in-process or multi-process) determines what kind of connection the driver gets.

## Get the DSN

```rust
use pglite::PGlite;

let db = PGlite::open("./mydata").await?;
let url = db.unix_uri().await?;
// url is a postgresql://postgres@localhost/postgres?host=/tmp/pgl-…/.s.PGSQL.5432
```

In multi-process mode, `unix_uri()` returns the child postmaster's native socket URI — the *same* DSN shape, just backed by a real Postgres server running in a child process. The driver code does not change.

In in-process mode, the first `unix_uri()` call lazily starts the embedded gateway. The gateway holds one session; ORM pool size must be 1. In multi-process mode, the gateway is bypassed and `unix_uri()` hands back the postmaster's URI; pool size is up to `MultiProcessOptions.max_connections`.

## SQLx

```rust
use sqlx::postgres::PgPoolOptions;

let url = db.unix_uri().await?;
let pool = PgPoolOptions::new()
    .max_connections(1)              // in-process: one session
    .connect(&url)
    .await?;

let row: (i32, String) = sqlx::query_as("SELECT id, name FROM users LIMIT 1")
    .fetch_one(&pool)
    .await?;
```

In multi-process mode, set `max_connections` to whatever `MultiProcessOptions.max_connections` is. Both sides agree on the pool ceiling.

## SeaORM

```rust
use sea_orm::{Database, DbConn};

let url = db.unix_uri().await?;
let conn: DbConn = Database::connect(&url).await?;

let rows: Vec<entity::user::Model> = entity::user::Entity::find()
    .all(&conn)
    .await?;
```

SeaORM wraps `sqlx`; the same `max_connections = 1` rule applies in in-process mode.

## Diesel

```rust
use diesel::pg::PgConnection;
use diesel::prelude::*;

let url = db.unix_uri().await?;
let mut conn = PgConnection::establish(&url)?;

let results = users::table
    .filter(users::id.gt(0))
    .load::<(i32, String)>(&mut conn)?;
```

Diesel manages its own connection; you pass a `&mut PgConnection` around or use `r2d2` if you want pooling. In in-process mode, keep one connection open at a time.

## tokio-postgres / postgres

```rust
use tokio_postgres::NoTls;

let url = db.unix_uri().await?;
let (client, connection) = tokio_postgres::connect(&url, NoTls).await?;

tokio::spawn(async move {
    if let Err(e) = connection.await {
        eprintln!("connection error: {e}");
    }
});

let row = client.query_one("SELECT id, name FROM users LIMIT 1", &[]).await?;
let name: &str = row.get(1);
```

The synchronous `postgres` crate (used by Diesel) follows the same shape: `postgres::Client::connect(&url, NoTls)?`.

## Notes that apply to every driver

- **The driver and the Rust API share a session in in-process mode.** A `db.exec` issued while the driver holds an open transaction runs *inside* that transaction. Sequencing is the caller's problem. If you want clean separation, use multi-process mode.
- **`COPY ... TO STDOUT` works** through the gateway. **`COPY ... FROM STDIN` does not** — the in-process engine cannot pause mid-COPY across roundtrips. For bulk loads, use multi-process mode or `db.copy_in` directly.
- **Connection pool size = 1** in in-process mode. The gateway accepts one client at a time. The doorman holds the single `Backend` and the in-process `tx_lock`. Set the pool to 1 even if the driver accepts more — the second connection will hang until the first closes.
- **Migrations.** All the standard Postgres migration tools (sqlx-cli, refinery, sea-orm-cli, diesel migrations) work because the engine is real Postgres. Just point the migration tool at the `unix_uri()` DSN.
- **PostgreSQL extensions.** The cargo features `pgvector`, `pgcrypto`, and `icu` enable extensions at build time. To use them, run `CREATE EXTENSION <name>` in a migration before the ORM sees the schema. The build script also needs `export_dynamic` set; see [Installation](/pglite-rs/installation#build-script-note-for-extensions-and-plpgsql).

## Multi-process mode

In multi-process mode, the DSN points at a real postmaster. Every rule above changes:

- **Pool size up to `MultiProcessOptions.max_connections`.** Each connection is a real Postgres backend with its own session state.
- **No shared-session surprises.** A `db.exec` does not collide with an open ORM transaction because they are different backends.
- **`COPY ... FROM STDIN` works.** The postmaster is a real Postgres, so COPY roundtrips are normal.
- **Latency.** Each query hops your process, the unix socket, the postmaster, and a backend. In-process mode is faster; multi-process is faster than the alternative (a real Postgres server on a different machine).

The driver code is the same. Just bump `max_connections` and turn on the `multiple-process` cargo feature.
