# Quickstart

The smallest end-to-end path: `init` against an upstream Postgres, `serve` PgPaw, send one snapshot query, then one live query. If you have done this once, the rest of the guides pick up a single concept at a time.

Assumes `pgpaw` is installed (see [Installation](/pgpaw/installation)) and a reachable Postgres 13+ with superuser access.

## 1. One-time upstream setup

```bash
pgpaw init \
    --pg-host 127.0.0.1 \
    --pg-user postgres \
    --pg-password "$POSTGRES_PASSWORD" \
    --pg-database myapp
```

`init` runs `ALTER SYSTEM` to set `wal_level = logical` and to raise `max_wal_senders` / `max_replication_slots` to `10` when they are below it, runs `CREATE PUBLICATION ... FOR ALL TABLES`, and installs the `pglite_ddl_watch` DDL event trigger (online schema-change capture). It does **not** create the replication slot ‚Äî that happens at serve time.

If `init` reports it changed `wal_level`, restart Postgres. The publication and DDL trigger are durable in the catalog ‚Äî `init` is a one-shot.

> Some internal help and error text still self-identifies as `cache-server` (a known internal mismatch). The published binary and the command you run is `pgpaw`, so if an error tells you to `Run cache-server init`, it means `pgpaw init`.

## 2. Serve

```bash
pgpaw serve \
    --pg-host 127.0.0.1 \
    --pg-user postgres \
    --pg-password "$POSTGRES_PASSWORD" \
    --pg-database myapp \
    --port 8080
```

On serve, `preflight` validates the upstream: `wal_level = logical`, `max_wal_senders` and `max_replication_slots` each `>= 1`, and that the publication exists ‚Äî erroring with remediation text otherwise. (Note the asymmetry: `init` *applies* the value `10`, while the serve-time `preflight` gate only requires `>= 1`.) PgPaw then opens the embedded pglite replica, **creates the replication slot at this point**, runs the initial backfill, and binds HTTP.

First boot runs the initial backfill; later boots resume from the watermark. The HTTP server is up while the replica catches up ‚Äî queries still work, they just see the current snapshot.

## 3. Wait for healthy

```bash
curl -s http://127.0.0.1:8080/healthz
# {"status":"ok","watermark":12345678}
```

A healthy replica returns `200` with `{"status":"ok","watermark":...}`. If the applier hits an unrecoverable error the replica is `halted`, and `/healthz` returns `503` with `{"status":"halted","reason":...}`. See [How it works](/pgpaw/how-it-works#why-the-replica-is-the-right-place-to-run-reads) for the failure modes.

## 4. Send a snapshot query

The request body accepts exactly one field, `sql`. No `params`, `role`, or extra keys are read.

```bash
curl -i -X POST http://127.0.0.1:8080/query \
     -H 'content-type: application/json' \
     -d '{"sql":"select id, email from users where id = 7"}'
```

You get a `303 See Other` with `Location: /q/{hash}/{version}` and `Cache-Control: no-store` on the redirect itself. Follow the redirect to fetch the body, which sets `ETag: {hash}:{version}` and `Cache-Control: public, max-age=259200` (72 hours):

```bash
curl -L -X POST http://127.0.0.1:8080/query \
     -H 'content-type: application/json' \
     -d '{"sql":"select id, email from users where id = 7"}'
# [{"id":7,"email":"alice@example.com"}]
```

That URL serves the same frozen body for its 72-hour cache window. Because the URL is content + version addressed, a relevant upstream write does not mutate it ‚Äî it re-issues a *new* `/q/{hash}/{version}` on the next `POST /query`.

## 5. Open a live stream

```bash
curl -N -X POST 'http://127.0.0.1:8080/query?live=true' \
     -H 'content-type: application/json' \
     -d '{"sql":"select id, status from orders"}'
```

The stream opens with one `snapshot` frame ‚Äî for a public query it carries a `url` pointer (`{"type":"snapshot","url":"/q/{hash}/{version}","version":...}`), for an access-controlled query it carries inline `rows` (`{"type":"snapshot","rows":[...],"version":...}`). Then, on each relevant upstream commit, the server re-runs the SQL, diffs against the last snapshot, and emits zero or more per-row frames ‚Äî `{"op":"insert","key":...,"row":...,"txid":...}`, `update`, or `delete` (delete carries `key` only) ‚Äî followed by a terminal `{"op":"up-to-date","txid":...}` for that same commit. The `up-to-date` frame is emitted even when the diff is empty, so an `awaitTxId` can't hang.

Every delta frame and the `up-to-date` frame carry `txid`, the low 32 bits of the upstream transaction xid. On a replication `Lagged` (dropped transactions under back-pressure), the server sends `{"op":"reset"}` and the client should reload. Frames are SSE-framed as `data: <json>\n\n`. See [API](/pgpaw/api/live) for the full event format.
