# Configuration

Worldant's user-facing configuration lives in the `WorldConfig` object you pass to `install()` (or `createWorld()`). The supervisor's scale-to-zero knobs are documented separately on the [Supervisor](/worldant/supervisor) page.

Configuration is **code-first**: every value flows from the `createWorld({ ... })` call in your app. There are no CLI config flags for any of these â€” the only supervisor flag that affects binding is `--public-host` on `start`/`run`.

## WorldConfig

`WorldConfig` is what you pass when you boot the world. All fields are optional.

```ts title="src/world.ts"
import { install } from '@midwess/worldant';

await install({
  dataDir: './data/world.db',
  deploymentId: 'prod-2024-11',
  id: 'checkout-app',
  config: { /* SupervisionConfig â€” see the Supervisor page */ },
});
```

| Field | Type | Default | What it does |
|---|---|---|---|
| `dataDir` | `string` | _none_ | Path of the SQLite database **file**. With it, the world is durable; omit it and the world is in-memory/ephemeral (see [dataDir](#datadir-durable-vs-ephemeral) below) |
| `deploymentId` | `string` | `dpl_<ULID>` | Deployment id surfaced back through `getDeploymentId()`. Set this from your CI's build id or git SHA |
| `id` | `string` | derived from `package.json` | The supervision id. Used to name the control socket and dedup lock on disk. See [Identity derivation](#identity-derivation) below |
| `config` | `SupervisionConfig` | `{}` | Scale-to-zero knobs. See [Supervisor](/worldant/supervisor#supervisionconfig) |

A note on `deploymentId`: this field only flows back through `getDeploymentId()`. It is **not** the per-run `deployment_id` stored on each `world_runs` row â€” that value comes from `@workflow/core`'s `RunCreated` event data, not from `WorldConfig.deploymentId`.

### dataDir: durable vs ephemeral

`dataDir` is the single biggest correctness decision on this page. It is the path of the SQLite database **file** itself â€” not a directory â€” and it is passed straight through to `SqliteConnectionManager::file(data_dir)`.

- **Set `dataDir`** â†’ the native open path calls `World::open(dataDir)`, which opens the file at exactly that path in WAL mode. Two sidecar files appear next to it: `<dataDir>-wal` and `<dataDir>-shm`. This database survives restarts, scale-to-zero, and redeploys.
- **Omit `dataDir`** â†’ the native open path calls `World::open_temp()`, which creates an **ephemeral in-memory** database (`file:world-mem-<n>?mode=memory&cache=shared`). It is lost the moment the process exits.

There is no `./.world` default, no directory creation, and no `:memory:` string option. The library does not create directories â€” the parent directory of `dataDir` must already exist (the example app creates its own). Passing the literal string `":memory:"` would be treated as a filename on disk, not as a special in-memory marker; to get an in-memory database you **omit** `dataDir`.

The example app â€” not the library â€” supplies its own default of `join(process.cwd(), '.worldant-data')` via the `WORLDANT_DATA_DIR` env var.

### Identity derivation

If you don't set `id`, worldant derives one from the calling app's `package.json`:

```
<sanitized-package-name>-<sha256("<packageName>:<appRoot>").slice(0, 16)>
```

The sanitizer replaces only characters **not** in `[a-zA-Z0-9._-]` with `-` â€” dots, underscores, and existing hyphens in the package name are preserved â€” then strips leading/trailing dashes (defaulting to `app` if the result is empty). The hash is a 16-character hex prefix. Note that the hash is computed over the **raw** package name (`<packageName>:<appRoot>`), not the sanitized one.

**You must set `id` explicitly when:**

- Your app has no `package.json` (e.g. a one-off script). `resolveWorldId` throws if it can't resolve both a package name and an entry path.
- You run multiple replicas on the same host. Each needs a unique `id` to avoid colliding on the control socket and dedup lock.
- You want the socket and lock paths to be stable across deploys. The derived id changes if `appRoot` moves (e.g. across `pnpm` symlinks); an explicit `id` pins it.

## Codec

The TS adapter encodes data as JSON across the NAPI boundary, with two extensions:

- `Uint8Array` values are encoded as `{ __u8b64: <base64> }` and decoded back to `Uint8Array`.
- `Date` values are encoded as `{ __date: <ISO string> }` and decoded back to `Date`.

This is internal â€” you do not need to think about it unless you are debugging a serialization issue across the FFI boundary. Plain JSON values (strings, numbers, booleans, objects, arrays, null) pass through unchanged.

## Paths on disk

When the world boots, two kinds of paths appear on disk:

| Path | Purpose | Lifecycle |
|---|---|---|
| `<dataDir>`, `<dataDir>-wal`, `<dataDir>-shm` | The SQLite database file plus its WAL sidecars. WAL mode, single writer, reader pool | Persistent. Survives restarts. Only present when `dataDir` is set â€” omit it and everything stays in memory |
| `$XDG_RUNTIME_DIR/worldant-<idHash>.*` | Control socket, dedup lock, private app socket. See [Supervisor](/worldant/supervisor#on-disk-paths-supervisor) | Created by the supervisor, removed on clean shutdown |

`dataDir` is yours, and it is a single file path â€” there is no enclosing directory the library creates. The runtime dir is shared with the OS: `$XDG_RUNTIME_DIR` when set (on Linux it is `/run/user/<uid>`), otherwise the library falls back to the OS temp dir (`os.tmpdir()`, usually `/tmp` on macOS).
